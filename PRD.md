# CampusLog PRD

## 1. 프로젝트 개요

CampusLog는 대학생이 프로젝트, 공모전, 인턴 등 다양한 활동 경험을 잊지 않게 기록하고, AI가 역량과 성과를 정리해 자기소개서·포트폴리오·면접 준비에 바로 활용할 수 있게 돕는 성장 기록 서비스입니다.

이 프로젝트는 커널아카데미 대학생 AI 크루 UNIKER 1기 팀 핏줄의 10주 MVP 개발 프로젝트입니다. **1차 MVP와 v1.1 제품 고도화는 완료된 제품 기준선(baseline)**이며, 현재 활성 계획 단계는 **2차 MVP**입니다. v1.1 변경은 main에 반영되었고, 2차 MVP 기능은 최신 main에서 별도 기능 브랜치로 시작합니다. 2차 MVP는 v1.1의 진행형 활동 기록과 CampusLog AI 흐름을 보존하면서 사용자 계정·영속 데이터 기반을 추가하고, AI 품질과 제품 디자인·사용자 경험을 함께 고도화합니다.

## 2. 해결하려는 문제

대학생들은 프로젝트, 공모전, 대외활동, 인턴, 수상, 블로그, GitHub 등 다양한 경험을 여러 곳에 기록하거나 아예 기록하지 않는 경우가 많습니다.

그 결과 자기소개서, 포트폴리오, 면접, 대외활동 지원 등을 준비할 때 자신의 경험을 다시 찾고 정리하는 데 많은 시간이 걸립니다. 시간이 지나면서 중요한 경험을 잊어버리거나, 어떤 경험을 어떤 상황에 활용해야 할지 판단하기 어려운 문제도 발생합니다.

CampusLog는 활동 중에는 매일 실제로 한 일을 가볍게 축적하고, 활동 종료 후에는 그 기록을 완성된 경험으로 정리하며, 필요할 때 AI가 목적에 적합한 경험과 활용 방향을 제안하도록 설계합니다.

## 3. 대상 사용자

CampusLog의 주요 사용자는 프로젝트, 공모전, 대외활동, 서포터즈, 인턴, 취업 등을 준비하는 대학생입니다.

- 활동 경험은 있지만 정리해둔 곳이 없는 대학생
- 활동이 끝난 뒤 경험을 한 번에 회상하는 데 부담을 느끼는 대학생
- 자기소개서, 포트폴리오, 면접을 준비하며 어떤 경험을 써야 할지 고민하는 대학생
- 자신의 활동 경험에서 역량, 성과, 키워드를 뽑아내는 데 어려움을 느끼는 대학생
- 여러 기기와 브라우저에서도 자신의 기록을 안정적으로 이어서 사용하려는 대학생

## 4. 사용 상황

사용자는 현재 또는 미래 활동의 제목과 간단한 정보를 먼저 등록하고, 캘린더에서 날짜별로 실제로 한 일을 기록합니다. 활동이 끝나면 해당 활동에 연결된 기록만 AI가 사실 기반 초안으로 합성하고, 사용자가 검토·수정한 뒤 완성된 경험으로 저장합니다. 이미 끝난 과거 경험은 기존 방식으로 직접 등록할 수 있습니다.

이후 자기소개서 문항, 포트폴리오 작성 목적, 면접 질문처럼 특정 상황이 생기면 CampusLog AI에 해당 내용을 입력합니다. AI는 저장된 경험과 분석 결과를 참고해 적합한 활동과 추천 이유, 활용 방향을 제공합니다.

## 5. 개발 단계와 현재 상태

### 5.1 v1.0 1차 MVP — 완료

- Next.js Full Stack, Browser localStorage, Next.js App Router API Routes, OpenAI API 기반 핵심 흐름 검증
- 과거 활동 경험 CRUD
- 경험별 AI 요약·핵심 역량·성과·키워드 분석
- 저장된 경험을 활용한 AI 기반 활동 추천과 추천 기록
- 로그인·회원가입·DB 없이 단일 브라우저에서 동작하는 제품 기준선 완성

### 5.2 v1.1 제품 고도화 — 완료, 2차 MVP 기준선

v1.1은 1차 MVP를 대체하는 현재 제품 기준선입니다. 다음 기능과 사용자 흐름은 2차 MVP에서도 회귀 없이 보존합니다.

- `/dashboard`의 캘린더 기반 `오늘의 기록`
- 현재·미래 활동의 간단 등록과 진행 상태 관리
- 활동 태그와 연결된 날짜별 한 일 작성·수정·삭제
- 활동 종료 후 연결된 기록만을 이용한 AI 완료 경험 합성
- 사용자의 합성 초안 검토·수정 후 기존 `Experience`로 저장
- `/experiences`의 `나의 활동`에서 완료 경험과 진행 활동 통합 확인
- `CampusLog AI` 내부의 `AI 기반 활동 추천`과 `추천 기록`
- 최신 검정·차콜 기반 CampusLog 제품 UI와 반응형 레이아웃

### 5.3 2차 MVP — 현재 활성 계획 단계

2차 MVP는 외부 사용자가 계정을 만들고 자신의 데이터를 안전하게 저장·조회할 수 있는 사용자 기반 제품을 완성하는 단계입니다. v1.1의 핵심 흐름을 유지하면서 아래 두 Track을 병행합니다.

#### Track A — 계정·데이터·AI 고도화

다른 팀원이 주로 담당하며, 변경 사항은 공통 데이터 계약과 화면 상태를 기준으로 함께 검토합니다.

- 로그인·회원가입·로그아웃 및 인증 세션 처리
- Supabase Auth 기반 사용자 식별
- Supabase Postgres 기반 사용자별 활동·일일 기록·완료 경험·AI 결과 저장
- 필요한 첨부 기능이 확정된 경우에 한해 Supabase Storage 연동
- AI 경험 분석·활동 추천의 품질, 안정성, 오류 처리 고도화
- 기존 완료 경험 합성의 인증·DB 전환 호환성과 사실성 유지
- AI 입력·출력 스키마, 프롬프트 버전, 결과 저장 규칙 정리

#### Track B — 디자인·사용자 경험 고도화

사용자(제품 오너)가 주로 담당하며, 인증·DB·AI 상태가 화면에서 자연스럽게 이어지도록 Track A와 인터페이스를 맞춥니다.

- 현재 CampusLog 브랜드 양식을 유지한 디자인 시스템 정교화
- 가입부터 활동 기록, AI 분석·추천까지의 온보딩과 핵심 동선 개선
- 로딩·빈 상태·오류·성공·권한 없음 상태의 일관된 UX
- 모바일·데스크톱 반응형, 키보드 접근성, 텍스트 대비, reduced motion 개선
- 사용자가 데이터 저장·AI 처리 상태와 다음 행동을 분명히 이해하도록 피드백 개선

두 Track은 별도 브랜치와 작은 PR로 진행하되, 데이터 타입·API 계약·공통 컴포넌트 변경은 사전에 공유합니다. 한 Track의 구현 편의를 위해 다른 Track의 사용자 경험이나 데이터 계약을 임의로 변경하지 않습니다.

### 5.4 UNIKER 이후 개인 Backend Portfolio Edition — 보류

Spring Boot REST API, MySQL 또는 AWS RDS, AWS S3, Spring Security/JWT는 CampusLog 2차 MVP 범위가 아닙니다. UNIKER 이후 별도 백엔드 포트폴리오 확장안에서 검토합니다.

## 6. 2차 MVP 목표

2차 MVP의 목표는 다음과 같습니다.

1. 사용자가 계정을 만들고 안전하게 로그인할 수 있습니다.
2. 사용자의 활동과 AI 결과가 계정에 귀속되어 다시 접속해도 유지됩니다.
3. v1.1의 진행형 활동 기록부터 완료 경험, AI 분석·추천까지의 핵심 흐름이 인증·DB 전환 후에도 동일하게 동작합니다.
4. AI 결과의 정확성·설명 가능성·실패 복구 경험을 개선합니다.
5. 처음 사용하는 사용자도 별도 설명 없이 핵심 가치를 경험할 수 있도록 디자인과 사용자 흐름을 고도화합니다.

## 7. 2차 MVP 포함 범위

### 7.1 인증과 사용자 계정

- 이메일 또는 이에 준하는 아이디 + 비밀번호 기반 회원가입·로그인·로그아웃
- Google OAuth 로그인
- Supabase Auth 세션 유지와 보호된 화면 접근 제어
- 인증 로딩, 만료, 실패, 재시도 상태 처리
- 최소한의 계정 정보와 개인정보 처리 안내

학교 인증, 복잡한 프로필 설정, 다수 소셜 provider는 별도 승인 없이는 포함하지 않습니다.

### 7.2 사용자별 데이터 저장

- `Experience`, `TrackedActivity`, `DailyLog`, 미완료 `ExperienceSynthesisDraft`, AI 분석 결과, 추천 기록의 사용자별 저장·조회·수정·삭제
- 모든 사용자 소유 데이터에 `user_id` 또는 이에 준하는 소유권 관계 적용
- 데이터 접근 정책과 Row Level Security 적용
- 합성 초안은 원본 활동과 사용된 날짜별 기록을 참조하고, 완료 경험 저장이 검증된 뒤 제거하는 v1.1 보존 정책을 기본값으로 사용
- 합성 실패·재시도 상태는 원본 활동에 연결하며, 합성 API에도 세션·소유권 검증·호출 제한과 완료 경험 생성 멱등성을 적용
- 저장 실패·동기화 중·재시도 상태를 사용자에게 명확히 표시
- 필요한 첨부 기능이 제품 범위로 확정된 경우에만 Supabase Storage 버킷·권한 정책 추가

### 7.3 AI 경험 분석·추천 고도화

- 구조화된 입출력 스키마와 검증 가능한 오류 처리
- 활동 원문에 없는 사실·수치·역할을 생성하지 않는 사실성 원칙 유지
- 분석과 추천 이유가 사용자의 원본 기록에서 어떻게 도출됐는지 이해할 수 있는 결과 구성
- 자기소개서 문항, 면접 질문, 지원서 화면 이미지 등에서 질문을 추출해 관련 경험을 추천하는 입력 확장
- JD / 직무 요구사항 / 우대사항 원문을 통째로 붙여넣으면 요구 역량을 구조화하고 보유 경험과 부족 경험을 비교하는 기능
- 추천된 경험을 바탕으로 지원 목적에 맞는 답변 초안 또는 활용 문장을 작성하는 기능
- AI 실패 시 원본 데이터 보존, 안전한 재시도, 중복 결과 생성 방지
- 프롬프트·모델·결과 형식의 버전 관리와 품질 비교 기준 마련
- OCR 이미지 입력은 우선 원본 파일을 저장하지 않는 일회성 AI 입력으로 설계하고, 원본 이미지 보존이 필요할 때만 Storage 범위를 별도 확정
- 구체적인 추천 후보 수, 비교 방식, 답변 초안 품질 기준은 사용자 검증 후 화면 명세에서 확정
- v1.1 완료 경험 합성은 새로운 고도화 범위를 임의로 늘리지 않고 인증·DB 연결 후의 사실성·멱등성·실패 복구를 회귀 검증

### 7.4 디자인과 사용자 경험

- 최신 CampusLog 검정·차콜 디자인 언어와 브랜드 일관성 유지
- 인증 전·후 정보 구조와 내비게이션 정리
- 활동 추가, 오늘 한 일 기록, 활동 종료, AI 분석·추천의 행동 우선순위 개선
- 모든 핵심 화면의 로딩·빈 상태·오류·성공 상태 설계
- 모바일·데스크톱 반응형과 접근성 기준 충족
- 과도한 시각 효과보다 기록과 AI 결과의 가독성, 다음 행동의 명확성을 우선

## 8. 2차 MVP 제외 범위

아래 항목은 사용자가 별도로 범위 변경을 승인하지 않는 한 2차 MVP에서 구현하지 않습니다.

- 학교 인증
- 커뮤니티와 팀 협업 기능
- 결제와 구독
- 관리자 페이지
- 공개 프로필
- GitHub, 블로그, Notion 등 외부 서비스 자동 연동
- PDF 포트폴리오 자동 생성
- 네이티브 모바일 앱
- 복잡한 조직·권한 체계
- 대규모 추천 알고리즘 또는 근거 없이 늘린 다수 추천 후보
- Spring Boot, MySQL/AWS RDS 등 개인 Backend Portfolio Edition 스택

파일 원본의 영구 저장과 Supabase Storage는 자동 포함하지 않습니다. OCR용 일회성 이미지 입력은 AI 고도화 후보에 포함하되, 수료증·활동사진·지원서 이미지 원본 저장은 실제 사용자 가치와 개인정보·비용 정책이 확정된 경우에만 작은 별도 작업으로 진행합니다.

## 9. 기술 전략

### 9.1 애플리케이션

- Next.js App Router + TypeScript
- Next.js Route Handler 또는 Server Action
- Tailwind CSS, shadcn/ui, lucide-react를 신규·리팩터링 UI에 선택적으로 사용
- 기존 CSS와 컴포넌트는 기능 변경과 무관하게 전면 교체하지 않음
- Vercel 배포

### 9.2 인증과 데이터

- Supabase Auth
- Supabase Postgres
- Row Level Security로 사용자 데이터 격리
- Supabase Storage는 승인된 첨부 기능에만 사용
- 서버에서 인증과 소유권을 재검증하며 클라이언트 입력의 `user_id`를 그대로 신뢰하지 않음

### 9.3 AI

- OpenAI API는 서버 측에서만 호출
- API Key는 서버 환경 변수로 관리하고 브라우저·저장소·로그에 노출하지 않음
- 구조화된 응답 검증, 제한 길이, 타임아웃, 재시도 및 오류 메시지 적용
- 개발·테스트에서 실제 API 호출은 명시적으로 필요한 경우에만 수행해 비용을 통제

## 10. 데이터 마이그레이션과 호환 원칙

v1.1의 Browser localStorage 데이터는 사용자의 기존 기록이므로 DB 전환 과정에서 묵시적으로 삭제하거나 덮어쓰지 않습니다.

- v1.1의 `Experience`, `TrackedActivity`, `DailyLog` 관계와 안정적인 식별자를 2차 MVP 데이터 모델의 출발점으로 사용합니다.
- 로그인 후 사용자가 로컬 기록을 계정으로 가져올지 명시적으로 선택할 수 있게 합니다.
- 마이그레이션은 동일한 로컬 항목을 여러 번 처리해도 중복 생성되지 않도록 멱등성을 보장합니다.
- 마이그레이션 완료를 확인하기 전에는 localStorage 원본을 자동 삭제하지 않습니다.
- 충돌이 발생하면 최신 데이터로 임의 덮어쓰지 않고 사용자에게 선택 가능하거나 복구 가능한 정책을 적용합니다.
- 샘플 데이터는 실제 사용자 데이터로 자동 이관하지 않습니다.
- AI 분석·합성·추천 결과에는 가능한 범위에서 프롬프트 또는 스키마 버전을 저장해 이후 형식 변경을 구분합니다.
- 기존 코드가 기대하는 도메인 타입과 화면 계약은 한 번에 파괴하지 않고 어댑터 또는 단계적 전환으로 호환합니다.
- 데이터 스키마 변경에는 마이그레이션 파일, 롤백 또는 복구 방안, 샘플 데이터가 아닌 테스트 데이터를 이용한 검증을 포함합니다.

## 11. 핵심 사용자 흐름

### 11.1 계정 진입

```text
서비스 진입
→ 회원가입 또는 로그인
→ 인증 성공
→ 로컬 기록이 있으면 가져오기 선택
→ 오늘의 기록
```

### 11.2 진행형 경험 기록

```text
오늘의 기록
→ 활동 추가
→ 활동 태그와 함께 날짜별 한 일 기록
→ 활동 종료
→ AI 완료 경험 초안 검토 / 수정
→ 나의 활동에 완료 경험 저장
```

### 11.3 과거 경험 기록과 AI 분석

```text
나의 활동
→ 과거 활동 경험 등록 또는 기존 활동 선택
→ AI 분석 요청
→ 요약 / 역량 / 성과 / 키워드 확인
→ 계정에 분석 결과 저장
```

### 11.4 CampusLog AI

```text
CampusLog AI
→ AI 기반 활동 추천
→ 자기소개서 / 포트폴리오 / 면접 질문 / JD 원문 / 지원서 이미지 입력
→ 입력에서 질문·요구사항·우대사항 추출
→ 저장된 경험과 분석 결과 기반 추천
→ 부족 경험과 보완 방향 비교
→ 추천 이유, 활용 방향, 답변 초안 확인
→ 추천 기록 저장 및 재확인
```

## 13. 2차 MVP 성공 기준

### 계정과 데이터

- 신규 사용자가 회원가입 후 로그인하고 세션을 유지할 수 있는가
- 인증되지 않은 사용자가 다른 사용자의 데이터에 접근할 수 없는가
- 활동·일일 기록·완료 경험·AI 결과가 사용자 계정에 정확히 귀속되는가
- 재접속 또는 다른 기기에서 로그인해도 저장된 데이터를 다시 확인할 수 있는가
- v1.1 로컬 데이터를 중복·손실 없이 선택적으로 가져올 수 있는가

### 핵심 기능과 AI

- v1.1의 진행형 활동 기록부터 완료 경험 저장까지의 흐름이 회귀 없이 동작하는가
- 과거 경험 입력, AI 분석, CampusLog AI 추천과 추천 기록이 DB 기반으로 이어지는가
- AI가 원본 기록에 없는 사실을 임의 생성하지 않고, 실패 시 원본을 보존하는가
- JD 원문 또는 질문 이미지처럼 긴 입력에서도 요구사항 추출, 경험 추천, 부족 경험 비교, 답변 초안이 근거 기반으로 생성되는가
- 중복 요청과 재시도에서 동일 활동 또는 추천 기록이 의도치 않게 중복 저장되지 않는가
- 팀이 정한 대표 시나리오에서 분석·추천 품질을 반복 비교할 수 있는가

### 디자인과 사용자 경험

- 처음 방문한 사용자가 가입 후 첫 활동을 기록하는 방법을 이해할 수 있는가
- 핵심 화면의 로딩·빈 상태·오류·성공 상태가 빠짐없이 제공되는가
- 모바일과 데스크톱에서 핵심 CTA, 폼, 캘린더, AI 결과가 깨지지 않는가
- 키보드 탐색, 포커스 표시, 텍스트 대비, reduced motion이 주요 흐름에서 보장되는가
- 계정·데이터·AI 처리 상태와 다음 행동을 사용자가 명확히 이해할 수 있는가

## 14. 보안·개인정보·비용 원칙

- API Key, 토큰, 비밀번호, 서비스 역할 키를 코드·문서·클라이언트 번들에 넣지 않습니다.
- `.env` 파일과 실제 사용자 데이터는 Git에 커밋하지 않습니다.
- 비밀번호를 직접 저장하지 않고 Supabase Auth를 사용합니다.
- Row Level Security와 서버 소유권 검증으로 사용자 데이터를 격리합니다.
- 수집하는 개인정보와 활동 데이터는 제품 기능에 필요한 최소 범위로 제한합니다.
- 로그에 사용자의 활동 원문, 인증 정보, AI 요청 전문을 무분별하게 남기지 않습니다.
- AI 요청에는 필요한 데이터만 전달하고, 보존·삭제 정책을 사용자에게 설명할 수 있어야 합니다.
- 실제 OpenAI API와 Storage 사용은 개발 환경에서도 비용과 호출량을 확인할 수 있게 관리합니다.
- 위험한 데이터 변경과 일괄 마이그레이션은 백업·복구 방안과 검증 없이 실행하지 않습니다.

## 15. 범위 변경 원칙

1차 MVP의 과거 제외 목록은 완료된 단계의 기록이며, 현재 2차 MVP 개발을 제한하지 않습니다. 현재 범위 판단은 이 PRD의 `2차 MVP 포함 범위`와 사용자의 최신 명시적 요구사항을 우선합니다.

다만 범위 확장이 보안·개인정보·비용·외부 서비스·데이터 손실 위험을 크게 바꾸는 경우에는 바로 구현하지 않고 팀이 영향과 성공 기준을 확인합니다. 승인된 변경은 PRD와 관련 화면·데이터 명세에 함께 반영합니다.
