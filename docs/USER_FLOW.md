# CampusLog User Flow

## 문서 상태

- 현재 제품 기준선은 개발이 완료된 `v1.1 제품 고도화`입니다.
- 다음 개발 단계는 `2차 MVP`이며, 이 문서는 v1.1의 동작을 보존하면서 2차 MVP에서 추가할 사용자 흐름을 정의합니다.
- `2차 MVP 목표`로 표시한 흐름은 구현 완료를 의미하지 않습니다. Supabase Auth / Postgres와 Google OAuth를 기본 기술로 사용하되 이메일 확인, 비밀번호 재설정, 추가 OAuth provider, 세션 세부 정책, AI 모델과 출력 형식은 담당 Track 검토 후 확정합니다.
- 1차 MVP 기록은 Git 이력과 별도 보관 문서에서 참고하며 현재 구현 판단 기준으로 사용하지 않습니다.

## 제품 흐름 원칙

1. v1.1의 `기록 → 활동 종료 → AI 정리 → 나의 활동 → AI 분석·추천` 흐름을 2차 MVP의 기준선으로 유지합니다.
2. 2차 MVP에서는 로그인한 사용자만 자신의 활동과 AI 결과에 접근할 수 있어야 합니다.
3. 브라우저에 남아 있는 v1.1 데이터는 로그인 도입 과정에서 임의로 삭제하거나 서버 데이터로 조용히 덮어쓰지 않습니다.
4. AI가 생성한 내용은 사용자 기록에 근거해야 하며, 실패·재시도·저장 상태를 사용자에게 명확히 보여줍니다.
5. 디자인·UX 개선은 핵심 정보 구조와 데이터 의미를 보존하면서 반응형, 접근성, 문구, 피드백과 작업 효율을 높이는 방향으로 진행합니다.

## v1.1 완료 기준선

### 1. 오늘의 기록과 진행 활동

```text
3D 브랜드 표지 (`/`)
→ 오늘의 기록 (`/dashboard`)
→ 활동 추가 버튼에서 확장되는 대형 둥근 작성 패널
→ 제목 / 간단한 내용 / 시작일 / 예상 종료일(날짜 또는 미정) 저장
→ 진행 중 또는 시작 예정 상태 확인
→ 캘린더에서 오늘이나 과거 날짜 선택
→ 날짜별 기록 헤더의 `+` 선택
→ 진행 활동이 없으면 필요 안내 팝업 확인 후 사용자가 선택할 때만 활동 추가
→ 검색 가능한 활동 Combobox에서 진행 활동 선택과 실제로 한 일 저장
→ 활동 제목·상태 중심의 날짜별 이벤트 목록 확인 / 수정 / 삭제
```

`/activities/new` 직접 진입과 새로고침은 같은 공용 활동 작성 폼을 사용하며, 저장 뒤 생성한 활동 상세로 이동합니다.

- 미래 날짜에는 한 일을 기록하지 않습니다.
- 활동 시작일보다 이른 날짜에는 해당 활동을 기록 대상으로 제공하지 않습니다.
- 완료된 활동의 날짜별 기록은 근거 데이터로 보존합니다.

### 2. 활동 종료와 완료 경험 생성

```text
진행 활동 상세 (`/activities/[id]`)
→ 활동 종료 확인
→ 연결된 날짜별 기록을 AI가 검토
→ 사실 기반 완료 경험 초안 생성
→ 사용자가 내용 / 성과 / 근거 부족 정보 확인 및 수정
→ 나의 활동에 완료 경험 저장
```

- 연결된 기록이 없으면 AI 정리를 시작하지 않고 기록 보완을 안내합니다.
- AI 호출 또는 저장이 실패하면 원본 활동과 날짜별 기록을 유지하고 재시도할 수 있게 합니다.
- 같은 활동에서 완료 경험이 중복 생성되지 않도록 연결 관계를 유지합니다.

### 3. 나의 활동과 과거 경험 직접 기록

```text
나의 활동 (`/experiences`)
├── 진행 중 활동 선택 → 진행 활동 상세
├── 완료 경험 선택 → 인라인 경험 상세
│   └── AI 분석 결과 선택 → 목록이 접힘 → 동일 폭의 왼쪽 활동 상세 + 오른쪽 분석 결과·보완 질문
└── 과거 경험 기록하기 (`/experiences/new`)
    → 전체 경험 입력
    → 경험 상세 / 수정 / 삭제
```

- 나의 활동은 진행 활동과 완료 경험을 한 목록에서 보여줍니다.
- 인라인 완료 경험 상세에는 `전체 화면으로 보기`를 두지 않습니다. 저장된 분석을 열면 같은 화면에서 상세와 분석을 비교하고, 분석을 닫으면 목록·상세 구조와 실행 버튼 초점을 복원합니다.
- 860px 이하에서는 활동 상세 다음에 분석 결과를 세로로 표시합니다. 직접 진입용 완료 경험 상세와 분석 경로는 유지합니다.
- 기존 과거 경험 직접 작성, 수정, 삭제 흐름을 유지합니다.

### 4. CampusLog AI

```text
나의 활동의 인라인 경험 상세
→ AI 경험 분석 요청
→ 저장된 AI 분석 결과 선택
→ 왼쪽 활동 상세 / 오른쪽 요약·STAR·근거·보완 질문 확인

CampusLog AI (`/recommend`)
→ 자기소개서 / 포트폴리오 / 면접 / JD / 대외활동·지원서 / 기타 활용 목적과 질문 입력
→ 저장된 완료 경험과 분석 결과를 바탕으로 활동 추천
→ 추천 이유 / 활용 방향 / 참고 문장 확인 및 복사
→ 추천 결과 자동 저장
→ 추천 기록 (`/recommend/history`)에서 다시 확인
```

- v1.1 추천은 가장 적합한 완료 경험 1개를 제시합니다.
- 추천 저장에 실패하면 성공 상태로 표시하지 않습니다.

## 2차 MVP 목표 흐름

### 1. 회원가입과 로그인

```text
비로그인 사용자
→ `/` 좌측 상단의 `CampusLog` 워드마크와 중앙 `대학생활을 기록하다.` 순환 문구 첫 화면
→ 작은 휠 입력 또는 진입 링크로 로그인·회원가입 viewport 이동
→ 회원가입은 이메일 또는 Google 선택
→ 이메일은 이메일·비밀번호 입력, Google은 OAuth로 해당 단계 생략
→ 이름 → 닉네임 Stepper 완료
→ 인증 세션과 비공개 프로필 저장
→ `/`의 3D 책 표지
→ 책 선택
→ 오늘의 기록 (`/dashboard`)
```

- 로그인은 이메일·비밀번호와 Google OAuth를 함께 제공합니다. 회원가입 첫 단계에서는 이메일과 Google을 선택하며, 이메일은 자격 증명 입력 뒤 이름·닉네임을 포함해 계정을 만들고 Google 신규·미완료 계정은 OAuth 시작 위치와 무관하게 callback 뒤 `/onboarding`에서 같은 이름·닉네임 단계를 완료합니다. 이메일 확인, 비밀번호 재설정과 추가 provider는 별도 결정 후 확장합니다.
- 이름·닉네임은 비공개 계정 개인화 metadata로만 저장하며 권한, RLS, 공개 프로필, 검색, 고유성 판단에는 사용하지 않습니다.
- 미완료 세션은 `/`, `/login`, `/signup`, 보호 화면에 직접 접근해도 `/onboarding`으로 돌아갑니다. 기존 계정에 완료 metadata가 없으면 최초 1회 이름·닉네임 입력을 요구하고, 온보딩 중 세션 만료 시 안전한 `returnTo`를 보존해 다시 로그인할 수 있게 합니다.
- 순환 문구의 명사는 `대학생활 / 공모전 / 해커톤 / 프로젝트 / 대회` 순서로 바뀌고, 조사는 문법에 따라 실제 값이 `을 ↔ 를`로 바뀌는 경우에만 전환합니다.
- `대학생활`은 주제 위계를 위해 진한 검정·강한 굵기로 유지하고 나머지 순환 명사는 옅고 가볍게 표시합니다. 조사 `을/를`은 항상 검정·강한 굵기로 고정하고, 글자 폭은 연속적으로 바뀌며 로그인 mode에는 별도 영문 eyebrow와 소개 문구를 두지 않습니다.
- `/login`과 `/signup`은 직접 URL, 보호 경로의 `returnTo`, 인증 오류 처리를 위한 fallback으로 유지합니다.
- 보호 화면에 비로그인 상태로 접근하면 로그인 화면으로 이동하고, 인증 후 책 표지를 다시 거치지 않고 원래 목적지로 돌아갈 수 있게 합니다.
- 세션 만료 시 재로그인을 안내하되 작성 중이거나 브라우저에 남아 있는 데이터를 즉시 삭제하지 않습니다.
- 로그아웃 후에는 다른 사용자가 이전 사용자의 서버 데이터를 볼 수 없어야 합니다.
- localStorage 데이터 확인과 이전 안내는 현재 기본 진입 흐름에 포함하지 않는 Deferred / Optional 기능입니다.

### 2. 사용자별 데이터베이스 저장

```text
로그인 사용자
→ 활동 / 날짜별 기록 / 완료 경험 생성·수정·삭제
→ 사용자 식별자와 함께 DB 저장
→ 다른 기기 또는 새 세션에서 같은 계정으로 조회
```

- 활동, 날짜별 기록, 완료 경험, AI 분석, 추천 기록을 사용자별로 분리합니다.
- 서버 저장 성공 여부를 UI에 반영하고, 실패했을 때 사용자의 입력을 잃지 않는 복구 경로를 제공합니다.
- 삭제 범위, 연관 데이터 처리, 동기화 충돌 정책은 데이터 스키마와 함께 확정합니다.

### 3. v1.1 localStorage 데이터 이전 (Deferred / Optional)

현재 정식 범위에서는 신규 사용자가 계정별 DB부터 시작하므로 아래 흐름을 필수로 구현하지 않습니다. localStorage 원본은 자동 이전하거나 자동 삭제하지 않으며, 로그인 세션에서는 계정 DB 데이터를 기본으로 사용합니다. 향후 실제 기존 사용자 데이터 보존 요구가 생기면 아래 흐름을 별도 승인 후 진행합니다.

```text
로그인 성공
→ 현재 브라우저의 v1.1 데이터 존재 여부 확인
→ 이전 대상과 영향을 설명
→ 사용자의 명시적 동의
→ DB 이전 진행 상태 표시
→ 성공 / 일부 실패 / 실패 결과 안내
→ 필요 시 재시도
```

- 서버 데이터와 로컬 데이터가 모두 있을 때 자동으로 한쪽을 덮어쓰지 않습니다.
- 이전 대상 개수에는 검증된 사용자 생성 데이터만 포함하고, 번들 샘플·개발 fixture·파싱 실패 항목은 제외 사유와 함께 구분합니다.
- 중복 판별, 충돌 해결, 이전 완료 후 localStorage 보관 기간은 구현 전에 팀이 확정해야 합니다.
- 이전을 미루거나 실패해도 로컬 원본은 검증 없이 삭제하지 않습니다.
- 네트워크 장애 시 임시 로컬 저장을 허용할지, 읽기 전용으로 제한할지 등 오프라인 fallback 정책은 별도 기술 결정 사항입니다.

### 4. AI 분석·추천 고도화

```text
로그인 사용자의 저장된 활동과 경험
→ 고도화된 AI 분석 또는 추천 요청
→ 요청 진행 상태
→ 근거가 연결된 결과
→ 사용자 확인 / 활용
→ 사용자별 DB에 결과 저장
→ 기록에서 재조회
```

- v1.1의 요약, 역량 태그, 성과, 키워드, 추천 이유, 활용 방향과 참고 문장은 최소 호환 기준입니다.
- 다음 AI 개발 순서는 AI API 보호, AI 분석 품질 개선, 목적 / JD / 질문 기반 추천, 부족 경험 비교와 답변 초안입니다.
- 추천 후보 수, 비교 방식, 개인화 기준, 피드백 학습 여부, 모델과 프롬프트 구성은 AI 담당 개발에서 확정합니다.
- 활동 종료 AI 합성은 계속 사용자의 확인과 수정을 거친 뒤 완료 경험으로 저장합니다.
- AI는 기록에 없는 성과를 사실처럼 만들지 않으며, 결과가 부족한 경우 부족한 근거를 안내합니다.
- AI 요청 제한, 비용 안내, 재생성 정책과 결과 버전 관리 방식은 운영 정책 확정 후 화면에 반영합니다.

## 인증 상태별 흐름

| 상태 | 접근 및 동작 |
| --- | --- |
| 비로그인 | `/`에서 순환 기록 문구와 중앙 인증 카드 표시. `/login`, `/signup` 접근 가능. 보호 화면은 로그인으로 연결 |
| Google 가입 후 프로필 미완료 | `/onboarding`에서 Google 이름을 수정 가능한 초기값으로 제안하고 이름·닉네임 저장 후 안전한 `returnTo`로 이동 |
| 로그인 | `/`에서 3D 책 표지 표시. 자신의 활동, 기록, 경험, AI 결과만 조회·변경 가능 |
| 인증 확인 중 | 보호 화면의 데이터를 먼저 노출하지 않고 일관된 로딩 상태 표시 |
| 세션 만료 | 재로그인 안내와 원래 목적지 복귀 경로 제공. 작성 데이터 보존 우선 |
| 네트워크 오류 | 오류 원인과 재시도 제공. 저장 성공으로 오인시키지 않음 |

## 개발 역할과 협업 경계

### 계정·데이터·AI 담당

- 로그인, 회원가입, 세션과 접근 제어
- 사용자별 DB 스키마, 저장 API, localStorage 자동 이전·자동 삭제 금지 정책. localStorage 이전 로직은 Deferred / Optional
- AI 분석·추천 API와 기존 활동 종료 합성 API의 인증·호출 제한·프롬프트·결과 스키마·저장 계약
- 합성 초안·실패·재시도 상태의 사용자 소유권, 보존 정책과 완료 경험 생성 멱등성
- 화면 개발자가 사용할 로딩·성공·빈 상태·오류 상태 계약 제공

### 디자인·UX 담당

- v1.1 화면의 시각적 고도화, 정보 위계, 반응형과 접근성 개선
- 인증, 동기화, AI 요청 상태의 사용자 안내와 상호작용 설계. 데이터 이전 UX는 Deferred / Optional
- 기존 핵심 흐름을 줄이지 않는 범위의 탐색 구조, 문구와 피드백 개선

### 공동 결정

- 화면에서 사용하는 데이터 필드나 API 응답 구조를 바꾸는 UX는 구현 전에 함께 합의합니다.
- 인증·DB 작업은 기존 화면을 임의로 재설계하지 않고 필요한 상태와 제약을 공유합니다.
- 데이터 충돌, 오프라인 fallback, AI 개인화와 추천 후보 방식처럼 아직 정하지 않은 항목은 별도 의사결정 기록 후 구현합니다.

## 2차 MVP 완료 판단

- 회원가입, 로그인, 로그아웃과 세션 만료 흐름을 끝까지 수행할 수 있습니다.
- 모든 보호 데이터가 사용자별로 분리되고 다른 계정에서 노출되지 않습니다.
- v1.1 로컬 데이터가 안내와 동의를 거쳐 안전하게 이전되거나 보존됩니다.
- 기존 오늘 기록, 활동 종료, 나의 활동, 분석과 추천 흐름이 DB 기반에서도 동작합니다.
- AI 고도화 결과가 근거, 실패 상태, 저장 상태와 함께 제공됩니다.
- 주요 화면이 모바일과 데스크톱에서 동작하고 키보드 탐색, 초점, 대비와 reduced motion 기준을 충족합니다.
