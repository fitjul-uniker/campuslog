# CampusLog User Flow

## 문서 상태

- 현재 제품 기준선은 개발이 완료된 `v1.1 제품 고도화`입니다.
- 다음 개발 단계는 `2차 MVP`이며, 이 문서는 v1.1의 동작을 보존하면서 2차 MVP에서 추가할 사용자 흐름을 정의합니다.
- `2차 MVP 목표`로 표시한 흐름은 구현 완료를 의미하지 않습니다. Supabase Auth / Postgres를 기본 기술로 사용하되 이메일 확인, 비밀번호 재설정, OAuth, 세션 세부 정책, AI 모델과 출력 형식은 담당 Track 검토 후 확정합니다.
- 1차 MVP 기록은 Git 이력과 별도 보관 문서에서 참고하며 현재 구현 판단 기준으로 사용하지 않습니다.

## 제품 흐름 원칙

1. v1.1의 `기록 → 활동 종료 → AI 정리 → 나의 활동 → AI 분석·추천` 흐름을 2차 MVP의 기준선으로 유지합니다.
2. 2차 MVP에서는 로그인한 사용자만 자신의 활동과 AI 결과에 접근할 수 있어야 합니다.
3. 브라우저에 남아 있는 v1.1 데이터는 로그인 도입 과정에서 임의로 삭제하거나 서버 데이터로 조용히 덮어쓰지 않습니다.
4. AI가 생성한 내용은 사용자 기록에 근거해야 하며, 실패·재시도·저장 상태를 사용자에게 명확히 보여줍니다.
5. 디자인·UX 개선은 핵심 정보 구조와 데이터 의미를 보존하면서 반응형, 접근성, 문구, 피드백과 작업 효율을 높이는 방향으로 진행합니다.

## v1.1 완료 기준선

### 1. 오늘의 기록과 진행 활동

```text
3D 브랜드 표지 (`/`)
→ 오늘의 기록 (`/dashboard`)
→ 활동 추가 (`/activities/new`)
→ 제목 / 간단한 내용 / 시작일 / 예상 종료일(날짜 또는 미정) 저장
→ 진행 중 또는 시작 예정 상태 확인
→ 캘린더에서 오늘이나 과거 날짜 선택
→ 진행 활동 선택
→ 실제로 한 일 저장
→ 날짜별 기록 확인 / 수정 / 삭제
```

- 미래 날짜에는 한 일을 기록하지 않습니다.
- 활동 시작일보다 이른 날짜에는 해당 활동을 기록 대상으로 제공하지 않습니다.
- 완료된 활동의 날짜별 기록은 근거 데이터로 보존합니다.

### 2. 활동 종료와 완료 경험 생성

```text
진행 활동 상세 (`/activities/[id]`)
→ 활동 종료 확인
→ 연결된 날짜별 기록을 AI가 검토
→ 사실 기반 완료 경험 초안 생성
→ 사용자가 내용 / 성과 / 근거 부족 정보 확인 및 수정
→ 나의 활동에 완료 경험 저장
```

- 연결된 기록이 없으면 AI 정리를 시작하지 않고 기록 보완을 안내합니다.
- AI 호출 또는 저장이 실패하면 원본 활동과 날짜별 기록을 유지하고 재시도할 수 있게 합니다.
- 같은 활동에서 완료 경험이 중복 생성되지 않도록 연결 관계를 유지합니다.

### 3. 나의 활동과 과거 경험 직접 기록

```text
나의 활동 (`/experiences`)
├── 진행 중 활동 선택 → 진행 활동 상세
├── 완료 경험 선택 → 경험 상세
└── 과거 경험 기록하기 (`/experiences/new`)
    → 전체 경험 입력
    → 경험 상세 / 수정 / 삭제
```

- 나의 활동은 진행 활동과 완료 경험을 한 목록에서 보여줍니다.
- 기존 과거 경험 직접 작성, 수정, 삭제 흐름을 유지합니다.

### 4. CampusLog AI

```text
경험 상세
→ AI 경험 분석 요청
→ 요약 / 역량 태그 / 성과 / 키워드 확인

CampusLog AI (`/recommend`)
→ 활용 목적과 질문 입력
→ 저장된 완료 경험과 분석 결과를 바탕으로 활동 추천
→ 추천 이유 / 활용 방향 / 참고 문장 확인 및 복사
→ 추천 결과 자동 저장
→ 추천 기록 (`/recommend/history`)에서 다시 확인
```

- v1.1 추천은 가장 적합한 완료 경험 1개를 제시합니다.
- 추천 저장에 실패하면 성공 상태로 표시하지 않습니다.

## 2차 MVP 목표 흐름

### 1. 회원가입과 로그인

```text
비로그인 사용자
→ 회원가입 (`/signup`) 또는 로그인 (`/login`)
→ 인증 성공 및 세션 생성
→ 로컬 데이터 확인
→ 필요한 경우 데이터 이전 안내
→ 오늘의 기록 (`/dashboard`)
```

- 이메일과 비밀번호를 기본 회원가입·로그인 입력으로 사용합니다. 이메일 확인, 비밀번호 재설정, OAuth 추가 여부는 별도 결정 후 확장합니다.
- 보호 화면에 비로그인 상태로 접근하면 로그인 화면으로 이동하고, 인증 후 원래 목적지로 돌아갈 수 있게 합니다.
- 세션 만료 시 재로그인을 안내하되 작성 중이거나 브라우저에 남아 있는 데이터를 즉시 삭제하지 않습니다.
- 로그아웃 후에는 다른 사용자가 이전 사용자의 서버 데이터를 볼 수 없어야 합니다.

### 2. 사용자별 데이터베이스 저장

```text
로그인 사용자
→ 활동 / 날짜별 기록 / 완료 경험 생성·수정·삭제
→ 사용자 식별자와 함께 DB 저장
→ 다른 기기 또는 새 세션에서 같은 계정으로 조회
```

- 활동, 날짜별 기록, 완료 경험, AI 분석, 추천 기록을 사용자별로 분리합니다.
- 서버 저장 성공 여부를 UI에 반영하고, 실패했을 때 사용자의 입력을 잃지 않는 복구 경로를 제공합니다.
- 삭제 범위, 연관 데이터 처리, 동기화 충돌 정책은 데이터 스키마와 함께 확정합니다.

### 3. v1.1 localStorage 데이터 이전

```text
로그인 성공
→ 현재 브라우저의 v1.1 데이터 존재 여부 확인
→ 이전 대상과 영향을 설명
→ 사용자의 명시적 동의
→ DB 이전 진행 상태 표시
→ 성공 / 일부 실패 / 실패 결과 안내
→ 필요 시 재시도
```

- 서버 데이터와 로컬 데이터가 모두 있을 때 자동으로 한쪽을 덮어쓰지 않습니다.
- 이전 대상 개수에는 검증된 사용자 생성 데이터만 포함하고, 번들 샘플·개발 fixture·파싱 실패 항목은 제외 사유와 함께 구분합니다.
- 중복 판별, 충돌 해결, 이전 완료 후 localStorage 보관 기간은 구현 전에 팀이 확정해야 합니다.
- 이전을 미루거나 실패해도 로컬 원본은 검증 없이 삭제하지 않습니다.
- 네트워크 장애 시 임시 로컬 저장을 허용할지, 읽기 전용으로 제한할지 등 오프라인 fallback 정책은 별도 기술 결정 사항입니다.

### 4. AI 분석·추천 고도화

```text
로그인 사용자의 저장된 활동과 경험
→ 고도화된 AI 분석 또는 추천 요청
→ 요청 진행 상태
→ 근거가 연결된 결과
→ 사용자 확인 / 활용
→ 사용자별 DB에 결과 저장
→ 기록에서 재조회
```

- v1.1의 요약, 역량 태그, 성과, 키워드, 추천 이유, 활용 방향과 참고 문장은 최소 호환 기준입니다.
- 추천 후보 수, 비교 방식, 개인화 기준, 피드백 학습 여부, 모델과 프롬프트 구성은 AI 담당 개발에서 확정합니다.
- 활동 종료 AI 합성은 계속 사용자의 확인과 수정을 거친 뒤 완료 경험으로 저장합니다.
- AI는 기록에 없는 성과를 사실처럼 만들지 않으며, 결과가 부족한 경우 부족한 근거를 안내합니다.
- AI 요청 제한, 비용 안내, 재생성 정책과 결과 버전 관리 방식은 운영 정책 확정 후 화면에 반영합니다.

## 인증 상태별 흐름

| 상태 | 접근 및 동작 |
| --- | --- |
| 비로그인 | `/`, `/login`, `/signup` 접근 가능. 보호 화면은 로그인으로 연결 |
| 로그인 | 자신의 활동, 기록, 경험, AI 결과만 조회·변경 가능 |
| 인증 확인 중 | 보호 화면의 데이터를 먼저 노출하지 않고 일관된 로딩 상태 표시 |
| 세션 만료 | 재로그인 안내와 원래 목적지 복귀 경로 제공. 작성 데이터 보존 우선 |
| 네트워크 오류 | 오류 원인과 재시도 제공. 저장 성공으로 오인시키지 않음 |

## 개발 역할과 협업 경계

### 계정·데이터·AI 담당

- 로그인, 회원가입, 세션과 접근 제어
- 사용자별 DB 스키마, 저장 API, localStorage 이전 로직
- AI 분석·추천 API와 기존 활동 종료 합성 API의 인증·호출 제한·프롬프트·결과 스키마·저장 계약
- 합성 초안·실패·재시도 상태의 사용자 소유권, 보존 정책과 완료 경험 생성 멱등성
- 화면 개발자가 사용할 로딩·성공·빈 상태·오류 상태 계약 제공

### 디자인·UX 담당

- v1.1 화면의 시각적 고도화, 정보 위계, 반응형과 접근성 개선
- 인증, 데이터 이전, 동기화, AI 요청 상태의 사용자 안내와 상호작용 설계
- 기존 핵심 흐름을 줄이지 않는 범위의 탐색 구조, 문구와 피드백 개선

### 공동 결정

- 화면에서 사용하는 데이터 필드나 API 응답 구조를 바꾸는 UX는 구현 전에 함께 합의합니다.
- 인증·DB 작업은 기존 화면을 임의로 재설계하지 않고 필요한 상태와 제약을 공유합니다.
- 데이터 충돌, 오프라인 fallback, AI 개인화와 추천 후보 방식처럼 아직 정하지 않은 항목은 별도 의사결정 기록 후 구현합니다.

## 2차 MVP 완료 판단

- 회원가입, 로그인, 로그아웃과 세션 만료 흐름을 끝까지 수행할 수 있습니다.
- 모든 보호 데이터가 사용자별로 분리되고 다른 계정에서 노출되지 않습니다.
- v1.1 로컬 데이터가 안내와 동의를 거쳐 안전하게 이전되거나 보존됩니다.
- 기존 오늘 기록, 활동 종료, 나의 활동, 분석과 추천 흐름이 DB 기반에서도 동작합니다.
- AI 고도화 결과가 근거, 실패 상태, 저장 상태와 함께 제공됩니다.
- 주요 화면이 모바일과 데스크톱에서 동작하고 키보드 탐색, 초점, 대비와 reduced motion 기준을 충족합니다.
