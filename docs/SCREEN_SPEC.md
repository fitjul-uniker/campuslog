# CampusLog Screen Specification

## 문서 상태

- 현재 화면 기준선은 구현이 완료된 `v1.1 제품 고도화`입니다.
- `2차 MVP 목표` 화면은 로그인·회원가입, 사용자별 DB 저장, AI API 보호와 AI 분석·추천 고도화를 위한 요구사항입니다. localStorage 데이터 이전 화면은 Deferred / Optional입니다.
- Supabase Auth / Postgres, 이메일·비밀번호와 Google OAuth 인증을 2차 MVP 기본 계약으로 사용합니다. 이메일 확인, 비밀번호 재설정, 추가 OAuth provider, AI 모델과 미확정 화면 구성은 별도 결정 전 구현된 것으로 보지 않습니다.
- 모든 화면은 기존 CampusLog의 검정·흰색 중심 브랜드, 넓은 여백과 절제된 표면 스타일을 이어가며 세부 시각 기준은 `DESIGN.md`를 따릅니다.

## 공통 화면 요구사항

### 앱 셸과 내비게이션

- `/`의 비로그인 순환 문구·인증 화면과 인증 후 3D 책 표지를 제외한 제품 화면은 공통 앱 셸을 사용합니다.
- 로그인 사용자의 주 내비게이션은 `오늘의 기록`, `나의 활동`, `CampusLog AI`입니다.
- 데스크톱 좌측 하단에는 얇은 구분선과 중립 표면 안에 원형 프로필 사진·닉네임을 표시하고, 계정 드롭다운에는 사용자 식별 정보와 로그아웃만 제공합니다. 모바일은 같은 메뉴를 44px 원형 아바타 진입점으로 축약합니다.
- 모바일에서 핵심 작업을 가리는 고정 요소를 두지 않고, 콘텐츠와 터치 영역이 화면 밖으로 넘치지 않게 합니다.

### 공통 상태

| 상태 | 화면 원칙 |
| --- | --- |
| 인증 확인 중 | 사용자 데이터를 먼저 노출하지 않고 일관된 로딩 상태 표시 |
| 초기 로딩 | 화면 구조를 예측할 수 있는 로딩 또는 상태 문구 제공 |
| 빈 상태 | 현재 상태의 이유와 다음 행동 1개 이상 제공 |
| 저장 중 | 중복 제출을 방지하고 작업 진행 중임을 표시 |
| 저장 성공 | 실제 서버 저장이 확인된 뒤 성공 피드백 제공 |
| 저장 실패 | 입력을 가능한 한 보존하고 오류와 재시도 경로 제공 |
| 권한 없음 | 다른 사용자의 데이터 내용을 노출하지 않고 재로그인 또는 접근 불가 상태 안내 |
| 네트워크 오류 | 성공으로 오인시키지 않고 재시도 또는 보존 상태 안내 |

### 접근성과 반응형

- 모든 입력은 보이는 레이블, 오류 연결, 키보드 조작과 명확한 `focus-visible`을 제공합니다.
- 색상만으로 성공·오류·비활성 상태를 전달하지 않고 문구나 의미 있는 구조를 함께 사용합니다.
- 상태 메시지는 필요한 경우 보조기술이 인지할 수 있는 라이브 영역으로 제공합니다.
- 모달이나 패널은 열릴 때 초점 이동, 초점 가두기, `Escape` 닫기와 닫힌 뒤 초점 복귀를 처리합니다.
- `prefers-reduced-motion`에서는 핵심 정보 이해에 필요하지 않은 이동, 부유와 반복 애니메이션을 줄입니다.

## 현재 구현 화면 기준선

### 1. 상태별 브랜드 진입 (`/`)

**목적**

비로그인 사용자에게 브랜드와 인증 진입을 먼저 제공하고, 인증된 사용자에게 검은 가죽 노트 브랜드 경험을 제공한 뒤 제품 화면으로 진입시킵니다.

**핵심 요소와 동작**

- 비로그인 상태에서는 좌측 상단에 `CampusLog` 워드마크를 고정 표시하고, 첫 viewport 중앙에 `대학생활을 기록하다.` 문구를 가장 먼저 표시합니다.
- 괄호나 네모난 단어 배경 없이 `대학생활을 / 공모전을 / 해커톤을 / 프로젝트를 / 대회를` 순서로 전환합니다. `대학생활`은 진한 검정과 강한 굵기, 나머지 순환 명사는 웜그레이와 한 단계 가벼운 굵기로 표시하되 조사 `을/를`은 항상 진한 검정·강한 굵기로 유지합니다. 기존 명사의 퇴장과 새 명사의 등장을 짧게 교차해 명사가 비는 프레임 없이 실제 글자 폭을 전환하며, 새 단어는 조사 쪽에 붙어 큰 폭 감소에서도 간격이 벌어지거나 기존 단어가 잘리지 않습니다. 조사는 실제 값이 `을 ↔ 를`로 바뀔 때만 별도로 전환하고 문장 전체는 매 상태와 전환 중에 화면 중앙을 유지합니다.
- 문구 아래에는 별도 `일시정지 / 재생` 컨트롤을 표시하지 않습니다. 명사와 조사 사이에는 2~5px 여유를 두고 reduced motion에서는 첫 문구로 고정합니다.
- 스크롤 시 제목은 진행도에 맞춰 부드럽게 사라지고, reduced motion에서는 첫 문구로 고정합니다.
- 첫 viewport의 하단 링크는 `스크롤하여 로그인 또는 회원가입` 한 줄과 약하게 움직이는 화살표만 표시합니다. 작은 아래 방향 휠 입력 또는 44px 이상 링크 선택 시 다음 viewport의 로그인·회원가입 영역으로 부드럽게 이동합니다. 인증 화면의 작은 위 방향 휠 입력은 첫 viewport로 자동 복귀시키지 않고 기본 스크롤 거리만 이동하며, 충분히 스크롤해야 첫 화면으로 돌아갑니다. reduced motion에서는 화살표 모션을 정지합니다.
- 인증 영역은 좌측 소개 문구와 구분선 없이 단일 카드를 중앙에 두고, 로그인과 회원가입 중 하나의 Primary CTA만 표시하며 같은 위치에서 mode를 전환합니다. 로그인 mode에서는 `Welcome back`과 제목 아래 소개 문구를 표시하지 않습니다.
- 이메일·비밀번호 등 인증 입력은 흰색 단일 표면, 1px 중립 border, shadow 없음으로 표시하고 focus-visible outline을 유지합니다.
- 인증 성공 후 `/`는 3D 노트를 표시하며, 노트는 장식이면서 `/dashboard` 진입 컨트롤입니다.
- 키보드 초점과 선택이 가능해야 하며, reduced motion에서는 포인터 추적과 반복 움직임을 줄입니다.
- 보호 화면에서 전달된 안전한 `returnTo`가 있으면 책 표지보다 원래 목적지 복귀를 우선합니다.

### 2. 오늘의 기록 (`/dashboard`)

**목적**

진행 활동을 확인하고, 캘린더 날짜에 실제로 한 일을 활동과 연결해 기록합니다.

**핵심 요소**

- 오늘 날짜와 `오늘의 기록` 제목
- 현재 진행 중인 활동: 메인 카드에는 활동 제목만 표시
- `활동 추가` 진입점
- 마무리 필요 활동과 시작 예정 활동
- 월간 기록 캘린더와 날짜별 기록 수
- 선택 날짜와 접근 가능한 `+` 기록 액션을 제공하는 날짜별 기록 이벤트 패널
- CTA 또는 기존 기록의 수정 버튼에서 여는 활동 선택·한 일 입력·저장 플로팅 패널
- 활동 제목·상태를 우선 표시하는 선택 날짜의 이벤트형 기록 목록과 수정·삭제 액션

**동작과 상태**

- `오늘의 기록`, `나의 활동`, `AI 기반 활동 추천` 상위 화면은 동일한 페이지 상단 여백, 제목 시작점, H1 크기와 설명 간격을 사용합니다. 세 화면 모두 `CampusLog > 현재 화면` Basic Breadcrumb을 H1 34px 위에 표시해 Breadcrumb과 제목 사이에 16px 시각 여백을 확보하며 제목 위치는 밀지 않습니다. 오늘 날짜는 `오늘의 기록` 제목 옆 메타 정보로 표시합니다.
- 활동 선택은 검색 입력을 받지 않는 읽기 전용 선택 필드로 제공합니다. 신규 기록 패널을 처음 열면 자동 선택 없이 `활동을 선택하세요`를 표시하고, 클릭 또는 방향키·Enter 선택과 행 오른쪽의 선택 체크 표시를 지원합니다. `Escape`는 열린 목록을 먼저 닫고 기록 패널은 유지합니다.
- 기록 패널 제목은 `한 일 남기기`, 입력 라벨은 `어떤 일을 하셨나요?`를 사용합니다. 패널은 데스크톱 최대 520px, 모바일은 좌우 14px 안전 여백을 제외한 유동 너비를 사용합니다. textarea는 최소 148px·기본 5줄 높이로 시작하고 긴 활동명은 Combobox 목록에서 최대 두 줄로 표시합니다.
- 미래 날짜에는 기록 폼을 제공하지 않습니다.
- 진행 활동이 없으면 `진행 활동 추가`, 진행 활동은 있지만 선택 날짜가 시작일보다 이르면 `오늘로 돌아가기`를 안내해 서로 다른 상태를 같은 행동으로 오인하지 않게 합니다.
- 상단과 날짜별 기록 빈 상태의 활동 추가 CTA는 누른 버튼에서 화면 가장자리 여백과 28px 둥근 모서리를 유지한 near-white 대형 작성 패널로 확장합니다. 패널에는 `어떤 활동을 기록할까요?`와 필수 폼만 표시하고, 닫을 때 원래 CTA 위치·크기·색으로 연속 축소합니다. 열릴 때 제목 입력으로 초점을 옮기고 배경 스크롤과 상호작용을 잠그며, `Escape`·닫기·취소와 초점 복귀, reduced motion을 지원합니다.
- 활동 상세에서 `activityId`를 전달해 진입하면 해당 활동을 선택한 상태로 기록 패널을 엽니다.
- 닫기 버튼, 배경 또는 `Escape`로 패널을 닫아도 작성 중 입력을 보존하고, `작성 취소`에서만 입력과 수정 상태를 초기화합니다.
- 저장 중에는 중복 제출을 막고 진행 상태를 표시합니다. 실패하면 입력과 열린 패널을 유지하며, 실제 저장 성공 후 패널을 닫고 카드의 라이브 상태로 결과를 알립니다.
- 패널은 열릴 때 textarea로 초점을 옮기고 내부에 초점을 가둔 뒤 닫힐 때 실행한 CTA 또는 수정 버튼으로 초점을 돌려줍니다.
- 기록 작성 패널은 visual viewport와 safe area 기준 화면 정중앙에 배치하고, 작은 화면에서는 내부 스크롤과 하단 footer로 저장 버튼 접근을 유지합니다.
- 별도 `오늘 한 일 기록하기` 카드는 표시하지 않습니다. 날짜별 기록 헤더의 `+`로 작성 패널을 열고, 목록에서는 연결 활동 제목과 기록 내용 한 줄을 기본 표시합니다. 앞쪽 세로 accent는 활동 상태와 관계없이 동일한 검정을 사용하며 진행 중·종료 상태는 목록에서 별도로 표시하지 않습니다. 항목 hover 또는 내부 키보드 초점에서는 카드 안에 내용을 최대 3줄로 펼치며 긴 내용은 말줄임 처리합니다. 전체 내용은 수정 패널에서 확인합니다.
- 캘린더 날짜 hover는 셀을 위로 이동시키지 않고 배경·테두리·글자색만 전환하며 선택 날짜와 오늘 표시의 우선순위를 유지합니다.
- 캘린더 헤더에는 별도 `월간 기록` 보조 문구를 표시하지 않고 현재 연·월과 이전 달·오늘·다음 달 탐색만 제공합니다.
- 진행 중인 활동이 없는 상태에서 `+`를 누르면 활동 작성 화면으로 즉시 전환하지 않고, 날짜별 기록에 진행 활동이 필요한 이유를 안내하는 팝업을 표시합니다. 사용자가 팝업의 `활동 추가`를 선택한 경우에만 활동 작성 화면을 엽니다.
- 작성 패널 도움말은 한 일을 자세히 적을수록 AI가 더 정확하게 분석할 수 있다고 안내합니다.
- 기록 수정과 삭제는 원래 날짜와 활동 연결을 유지하며 결과를 명확히 알립니다.
- `/activities/new`, `/activities/[id]`, `/experiences/new`, `/experiences/[id]`, `/experiences/[id]/edit`, `/experiences/[id]/analysis`, `/recommend/history`는 `CampusLog`부터 기능 상위 링크와 현재 위치까지 보여주는 Basic Breadcrumb을 H1 시작점 34px 위에 표시합니다. `CampusLog`는 브랜드 Petrona 800, 나머지는 UI 서체를 사용합니다. AI 추천 기록은 `CampusLog > AI 기반 활동 추천 > 추천 기록`으로 표시하며, 단계가 늘어나도 H1 좌표와 타이포그래피는 바뀌지 않습니다.

### 3. 활동 추가 (`/activities/new`)

**목적**

앞으로 진행하거나 현재 진행 중인 활동의 최소 정보를 등록합니다.

**입력**

- 활동 제목
- 간단한 내용
- 시작일
- 예상 종료일: 날짜 또는 `미정`

**동작과 상태**

- 미래 시작일은 시작 예정, 오늘 또는 과거 시작일은 진행 중으로 등록합니다.
- 예상 종료일은 시작일보다 빠를 수 없습니다.
- 대시보드에서는 대형 둥근 확장 작성 패널로 열고 저장 CTA를 아이콘과 `저장`으로 표시하며, 직접 URL·새로고침을 위한 `/activities/new` 페이지도 같은 공용 폼과 validation을 사용합니다.
- 대시보드의 `저장` 또는 직접 페이지의 `활동 저장` 성공 후 생성한 활동 상세로 이동합니다.
- 필수값, 날짜 관계와 저장 오류를 필드 또는 폼 수준에서 안내합니다.

### 4. 진행 활동 상세 (`/activities/[id]`)

**목적**

활동 상태, 기간과 날짜별 기록을 확인하고 활동 시작·종료와 완료 경험 생성을 수행합니다.

**핵심 요소**

- 제목, 상태, 활동 기간, 기록된 날과 누적 기록
- 활동 타임라인
- 시작 예정 활동의 시작 액션
- 진행 활동의 `오늘 한 일 기록하기`, `활동 종료`
- AI 처리, 실패, 초안 검토와 완료 경험 저장 상태

**활동 종료 흐름**

1. 종료 확인에서 연결 기록 수와 AI 초안 생성 사실을 설명합니다.
2. 기록이 없으면 종료하지 않고 기록 보완을 안내합니다.
3. AI는 날짜별 기록을 근거로 활동 내용, 성과와 근거 부족 정보를 정리합니다.
4. 사용자가 초안을 확인·수정한 뒤 `나의 활동`에 저장합니다.
5. 실패 시 활동 다시 열기 또는 AI 재시도를 제공하고 원본 기록을 보존합니다.

### 5. 나의 활동 (`/experiences`)

**목적**

진행 활동과 완료 경험을 한 목록에서 탐색합니다.

**핵심 요소와 동작**

- 전체 활동 수와 진행 중 활동 수
- 제목 검색
- `나의 활동` 화면 제목은 좁은 폭에서도 가로 한 줄을 유지합니다. 목록 패널이 좁아지면 검색을 아이콘으로 축약하고 최소 폭에서는 활동 수 배지를 숨깁니다. 모바일에서 닫힌 검색은 제목 행 오른쪽의 44px 아이콘으로 고정합니다. 펼칠 때도 제목은 유지하되 활동 수와 `진행 중` 배지만 잠시 숨기고, 같은 헤더 높이에서 제목 오른쪽의 남은 폭 안에 최대 250px 검색창을 표시합니다.
- 진행 활동의 `진행 중` 상태 표시
- 선택 후 목록을 유지하는 상세 패널
- 완료 경험 상세의 `전체 화면으로 보기`는 표시하지 않습니다. 저장된 `AI 분석 결과`를 선택하면 같은 `/experiences` 화면에서 목록을 접고 활동 상세를 왼쪽, 분석 결과와 기록 보완 질문을 오른쪽에 1:1 동일한 폭으로 표시합니다.
- 분석 스플릿뷰를 닫으면 기존 목록·상세 구조로 돌아가고 실행한 `AI 분석 결과` 버튼으로 초점을 복귀합니다. `Escape`는 분석 패널을 먼저 닫고, 분석 패널이 없을 때 활동 상세를 닫습니다.
- 과거 경험 기록 진입점
- 진행 활동은 `/activities/[id]`, 완료 경험은 `/experiences/[id]`로 연결합니다.
- 모바일에서는 목록 위, 상세 아래 구조를 사용하며 분석을 열면 활동 상세 다음에 분석 결과를 세로로 배치합니다. 상세를 닫으면 선택 항목으로 초점을 돌려줍니다.

### 6. 과거 경험 작성·수정 (`/experiences/new`, `/experiences/[id]/edit`)

**목적**

이미 끝난 프로젝트, 공모전, 인턴과 대외활동 경험을 직접 작성하거나 수정합니다.

**입력**

- 제목, 기간, 역할, 내용, 성과
- 관련 링크 URL과 설명

**동작과 상태**

- 작성·수정 화면에서는 AI 분석을 실행하지 않고 경험 저장에 집중합니다.
- 링크 형식, 빈 URL에 입력된 설명과 중복 URL을 검증합니다.
- 저장 후 완료 경험 상세로 이동하며, 기존 분석과 원본이 달라지면 재분석 필요 상태를 표시할 수 있습니다.

### 7. 완료 경험 상세 (`/experiences/[id]`)

**목적**

경험 원본과 분석 상태를 확인하고 수정·삭제·AI 분석으로 이동합니다.

**핵심 요소와 동작**

- 기간, 역할, 내용, 성과와 관련 링크
- 수정, 삭제, AI 분석 요청 또는 분석 결과 액션
- `/experiences` 인라인 상세의 분석 결과 액션은 별도 라우트 이동 대신 스플릿뷰를 열고, 직접 URL로 진입한 `/experiences/[id]`는 기존 상세 fallback을 유지합니다.
- 삭제 전 확인을 거치며 연결 데이터 처리 정책은 2차 MVP DB 설계와 일치시킵니다.

### 8. AI 경험 분석 (`/experiences/[id]/analysis`)

**목적**

특정 완료 경험의 AI 분석 결과를 보여줍니다.

**최소 출력 계약**

- 경험 요약
- 핵심 역량 태그
- 주요 성과
- 활용 가능한 키워드
- 분석 생성일

**동작과 상태**

- `/experiences`에서 저장된 결과를 선택하면 같은 화면의 오른쪽 분석 패널에 이 출력과 기록 보완 질문을 표시합니다. `/experiences/[id]/analysis`는 직접 진입과 기존 링크 호환을 위해 유지합니다.
- 미분석, 분석 중, 분석 완료, 재분석 필요와 실패 상태를 구분합니다.
- 다시 분석할 수 있고, 실패 시 원본 경험과 마지막으로 저장된 유효 결과를 임의로 삭제하지 않습니다.
- CampusLog AI의 활동 추천으로 이동할 수 있습니다.

### 9. CampusLog AI - AI 기반 활동 추천 (`/recommend`)

**목적**

활용 목적과 질문을 기준으로 완료 경험을 추천하고 추천 이유와 활용 문장을 제공합니다.

**입력과 최소 출력 계약**

- 입력: 자기소개서 / 포트폴리오 / 면접 / JD / 대외활동·지원서 / 기타 활용 목적, 질문 또는 문항
- 출력: 추천 경험, 추천 이유, 관련 역량 태그, 강조할 성과, 활용 방향, 참고 문장

**동작과 상태**

- v1.1에서는 가장 적합한 경험 1개를 추천합니다.
- 헤더의 `추천 기록` 액션으로 이전 결과를 확인할 수 있으며 현재 추천 화면을 반복하는 활성 탭이나 전역 내비게이션과 중복되는 `나의 활동` 바로가기는 표시하지 않습니다.
- 헤더는 `AI 기반 활동 추천` 제목과 `적합한 경험 Top 3를 근거와 함께 비교합니다.` 설명부터 시작하며 작은 `CampusLog AI` eyebrow를 반복 표시하지 않습니다.
- 추천 입력 패널은 저장 개수를 나열하지 않고 `저장된 활동과 분석 결과를 함께 참고합니다.`로 안내합니다.
- 결과 생성 성공 시 자동 저장하고, 저장 실패 시 성공 결과로 표시하지 않습니다.
- 실행 CTA는 `AI 분석`으로 표시하고 colorful Border Beam으로 AI 실행임을 구분합니다.
- 참고 문장은 44px 아이콘 전용 복사 버튼으로 복사하고 `복사` / `복사 완료` 접근성 이름, 성공 status, 실패 안내를 유지합니다.
- 완료 경험은 없지만 진행 활동이 있으면 오늘의 기록에서 먼저 완료 경험으로 정리하도록 안내합니다.

### 10. CampusLog AI - 추천 기록 (`/recommend/history`)

**목적**

이전 추천의 활용 목적, 질문, 추천 경험과 활용 문장을 다시 확인합니다.

**핵심 요소와 동작**

- 헤더의 `새 추천 받기` 액션으로 추천 입력 화면에 돌아갈 수 있습니다.
- 좌측 메뉴와 중복되는 `CampusLog AI` eyebrow는 표시하지 않습니다.
- 추천 기록 개수와 검색
- 목록을 유지한 추천 상세
- 추천에 사용된 원본 활동 이동과 문장 복사
- 기록이 없거나 검색 결과가 없을 때 서로 다른 빈 상태와 다음 행동을 제공합니다.

## 2차 MVP 목표 화면

### 11. 로그인 (`/` 인증 영역, `/login` fallback)

**목적**

기존 계정의 세션을 만들고 보호 화면 또는 원래 목적지로 이동합니다.

**필수 UI**

- CampusLog 브랜드와 로그인 제목
- 이메일과 비밀번호 입력
- 로그인 Primary CTA
- 회원가입 전환 링크
- 인증 중, 입력 오류, 인증 실패, 요청 제한과 네트워크 오류 상태

**동작 원칙**

- 입력값이나 계정 존재 여부를 오류 문구로 과도하게 노출하지 않습니다.
- 잘못된 자격 증명·가입 실패 같은 인증 오류는 Supabase 원문을 그대로 노출하지 않고, 계정 열거를 막는 공통 사용자용 오류 코드와 문구로 매핑합니다.
- 성공 시 안전한 보호 화면 `returnTo`가 있으면 해당 경로로, 없으면 `/`의 3D 책 표지로 이동합니다.
- 이미 로그인한 사용자가 `/login`에 접근하면 안전한 `returnTo` 또는 `/`의 책 표지로 연결합니다.
- Google OAuth 외 추가 소셜 로그인, 비밀번호 재설정과 이메일 확인 UI는 세부 정책 확정 전 추가하지 않습니다.

### 12. 회원가입 (`/` 인증 영역, `/signup` fallback)

**목적**

새 계정을 만들고 필요한 인증 절차를 거쳐 CampusLog를 시작합니다.

**필수 UI**

- 첫 단계의 `이메일로 회원가입`, `Google로 회원가입` 선택
- 이메일 선택 시에만 이메일과 비밀번호 입력
- 이메일·Google 공통 이름 → 닉네임 2단계 Stepper
- 이름은 1~40자, 닉네임은 1~20자이며 앞뒤 공백을 제거하고 연속 공백을 정규화
- 이름의 계정 식별·개인화 목적과 닉네임의 비공개 사용 범위 안내
- 약관 동의가 필요한 경우의 명확한 필수·선택 구분
- 회원가입 Primary CTA
- 로그인 전환 링크
- 단계별 이전·다음, 제출 중, 입력 유효성 오류, 사용자용 가입 실패, 인증 대기와 성공 상태

**동작 원칙**

- 수집하는 개인정보와 사용 목적을 필요한 범위에서 안내합니다.
- 이메일 가입은 자격 증명을 클라이언트 메모리에서 단계 이동 동안만 보존하고 URL, localStorage, DOM hidden input, 로그에는 저장하지 않습니다. 최종 제출 직전에만 `FormData`에 넣어 이름·닉네임과 함께 Supabase signup 요청을 보내며 서버 응답 후 비밀번호를 비웁니다.
- Google OAuth callback은 로그인·가입 query가 아니라 교환된 실제 계정의 완료 metadata를 확인합니다. 미완료 계정은 `/onboarding`으로 이동하고, provider가 준 이름은 수정 가능한 초기값으로만 사용하며 닉네임은 사용자가 직접 확인·입력합니다.
- 이름·닉네임은 `user_metadata.campuslog_profile`에 비공개 최소 계정 정보로 저장합니다. 사용자 수정 가능 metadata이므로 RLS·권한 판단에는 사용하지 않으며 닉네임 고유성·검색·공개 프로필은 제공하지 않습니다.
- 미완료 계정은 온보딩에서 이탈해 `/`, 인증 fallback, 보호 화면에 접근해도 다시 `/onboarding`으로 이동합니다. 기존 계정에 완료 metadata가 없으면 최초 1회 입력을 요구하고, 온보딩 중 세션 만료 오류에는 `returnTo`를 보존한 재로그인 CTA를 제공합니다.
- 가입 실패도 계정 존재 여부를 직접 확인해주지 않는 공통 사용자용 오류 code와 문구를 사용합니다.
- 가입 직후 세션을 만들지 추가 인증을 요구할지는 인증 정책에서 확정합니다.
- 가입 즉시 세션과 프로필 저장이 완료되면 안전한 `returnTo` 또는 `/`의 3D 책 표지로 이동하고, 이메일 확인이 필요하면 확인 전까지 책 표지를 노출하지 않습니다.
- 각 단계 변경 후 현재 단계에 초점을 이동하고, 오류 시 첫 invalid 입력에 초점을 둡니다. 320px 폭과 200% 확대에서 가로 잘림 없이 입력값과 오류 문구를 유지합니다.

### 13. 로컬 데이터 이전 안내 (Deferred / Optional)

현재 정식 범위에서는 신규 사용자가 계정별 DB부터 시작하므로 이 화면을 필수로 구현하지 않습니다. localStorage 원본은 자동 이전하거나 자동 삭제하지 않으며, 로그인 세션에서는 계정 DB 데이터를 기본으로 사용합니다.

**표시 조건**

향후 실제 기존 사용자 데이터 보존 요구가 생기고 이전 기능을 별도 승인한 경우에만 표시합니다.

**필수 정보와 액션**

- 발견된 데이터 종류와 개수
- 번들 샘플·개발 fixture·파싱 실패 항목을 제외한 실제 이전 가능 개수와 필요 시 제외 사유
- 현재 계정으로 이전된다는 설명
- 서버 데이터가 있을 때 자동 덮어쓰지 않는다는 안내
- 사용자의 명시적 이전 시작 액션
- 나중에 결정할 수 있는 비파괴적 보류 경로
- 진행률 또는 현재 처리 단계
- 성공, 일부 실패, 전체 실패와 재시도 결과

**안전 원칙**

- 안내 없이 자동 이전하거나 로컬 데이터를 삭제하지 않습니다.
- 이전 성공 여부를 항목별로 검증하기 전 원본을 제거하지 않습니다.
- 중복·충돌 해결 방식과 이전 완료 뒤 보관 기간은 DB 담당과 합의 후 문구와 액션을 확정합니다.
- 오프라인 fallback을 지원할지 결정되기 전에는 로컬 저장이 자동 동기화된다고 표현하지 않습니다.

### 14. 세션 만료와 재인증 상태

**필수 동작**

- 세션이 만료되면 이유와 재로그인 필요성을 알립니다.
- 현재 경로를 안전하게 보존해 로그인 후 복귀할 수 있게 합니다.
- 작성 중 입력은 가능한 한 브라우저 메모리나 합의된 fallback에 보존하고 즉시 폐기하지 않습니다.
- 보존할 수 없는 민감 정보는 예외이며, 어떤 정보가 사라지는지 제출 전에 안내합니다.

## AI 분석·추천 고도화 화면 원칙

- 다음 AI 기능 순서는 AI API 보호, AI 분석 품질 개선, 목적 / JD / 질문 기반 추천, 부족 경험 비교와 답변 초안입니다.
- 기존 분석과 추천의 최소 출력 계약을 제거하거나 의미를 바꿀 때는 화면, API 타입과 기존 저장 데이터 호환 계획을 함께 갱신합니다.
- 분석·추천 요청에는 `대기 → 처리 중 → 성공 또는 실패 → 재시도` 상태가 있어야 합니다.
- 결과에는 어떤 활동 또는 기록을 근거로 했는지 사용자가 이해할 수 있는 연결 정보를 제공합니다.
- 활동 종료 AI 초안은 자동으로 완료 경험에 확정하지 않고 계속 사용자 확인과 수정을 거칩니다.
- AI가 판단할 근거가 부족하면 내용을 지어내지 않고 부족한 정보를 표시합니다.
- 고도화 결과는 로그인 사용자 소유의 DB 데이터로 저장하며, 저장이 확인되기 전 완료 상태로 표시하지 않습니다.
- 호출 제한·입력 상한·중복 요청 오류는 공통 사용자용 error code로 처리하고, `retryAfter`가 있으면 재시도 가능 시점을 표시하며 입력을 보존합니다.
- 활동 종료 합성에도 같은 인증·호출 제한을 적용하고, 초안·실패·재시도 상태를 원본 활동 소유자에게만 표시합니다.
- 추천 후보 수, 비교 UI, 개인화, 사용자 피드백, 스트리밍, 생성 중 취소와 결과 버전 표시는 AI 담당의 계약이 확정된 뒤 설계합니다.

## 디자인·UX 작업 경계

### 디자인·UX 담당이 주도하는 항목

- v1.1 전 화면의 정보 위계, 간격, 타이포그래피, 표면과 인터랙션 고도화
- 데스크톱·모바일 반응형, 키보드 탐색, 초점, 대비와 reduced motion
- 로그인, 회원가입, 데이터 이전, 저장·동기화와 AI 상태의 이해 가능한 문구와 피드백
- 사용성 개선을 위한 탐색 구조 제안과 검증

### 계정·데이터·AI 담당이 제공해야 하는 계약

- 인증 상태, 세션 만료, 권한 오류와 경로 보호 동작
- 데이터별 조회·저장·삭제·마이그레이션 상태와 오류 코드
- AI 입력 한도, 처리 상태, 출력 필드, 근거, 저장, error code, `retryAfter`와 재시도 정책

### 공동 검토가 필요한 변경

- 화면에서 사용하는 데이터 필드와 API 응답 변경
- 인증 경로, 보호 범위와 `returnTo` 동작 변경
- 로컬·서버 충돌 해결, 오프라인 fallback과 삭제 정책
- AI 추천 후보 수나 결과 구조처럼 화면 레이아웃을 크게 바꾸는 기능

각 담당은 자신의 작업을 위해 다른 영역을 임의로 확정하지 않습니다. 미확정 항목은 로딩·빈 상태·오류 상태 계약과 함께 먼저 문서화한 뒤 구현합니다.

## 2차 MVP 화면 완료 기준

- 로그인, 회원가입, 로그아웃, 보호 경로와 세션 만료 상태를 모바일·데스크톱에서 완료할 수 있습니다.
- 로컬 데이터가 있는 사용자는 명시적 안내와 동의 후 이전하고 실패 시 재시도할 수 있습니다.
- 저장 성공·실패와 동기화 상태를 사용자가 구분할 수 있으며 입력을 불필요하게 잃지 않습니다.
- 기존 오늘 기록, 활동 종료, 나의 활동, AI 분석과 추천 화면이 사용자별 DB 데이터로 동작합니다.
- AI 고도화 결과가 근거와 저장 상태를 포함하고 기존 핵심 결과와 호환됩니다.
- 인증과 데이터 접근을 포함한 주요 흐름이 키보드, 초점, 대비와 reduced motion 기준을 충족합니다.
