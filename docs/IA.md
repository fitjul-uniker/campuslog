# CampusLog Information Architecture

## 문서 상태

- `v1.1 제품 고도화`의 IA는 구현 완료된 기준선입니다.
- `2차 MVP 목표`에는 인증, 사용자별 DB 저장과 AI 분석·추천 고도화를 추가합니다.
- 아래에서 `[현재]`는 v1.1에 구현된 경로, `[목표]`는 2차 MVP에서 추가하거나 보호 정책을 적용할 경로입니다.
- Supabase Auth / Postgres는 2차 MVP 기본 기술입니다. 이메일 확인, 비밀번호 재설정, OAuth, 마이페이지와 세부 AI 기능은 아직 확정하지 않습니다.

## 전체 사이트맵

```text
CampusLog
├── 공개 영역
│   ├── [현재] 3D 브랜드 표지 (`/`)
│   ├── [목표] 로그인 (`/login`)
│   └── [목표] 회원가입 (`/signup`)
└── [2차 MVP 목표] 인증 보호 영역
    ├── [현재] 오늘의 기록 (`/dashboard`)
    │   ├── 진행 중 / 시작 예정 활동 요약
    │   ├── 월간 기록 캘린더
    │   ├── 활동별 한 일 기록
    │   └── 날짜별 기록 확인 / 수정 / 삭제
    ├── [현재] 활동 추가 (`/activities/new`)
    ├── [현재] 진행 활동 상세 (`/activities/[id]`)
    │   ├── 활동 상태와 날짜별 기록
    │   ├── 활동 시작 / 종료 / 다시 열기
    │   ├── AI 완료 경험 합성
    │   └── 사용자 확인 후 완료 경험 저장
    ├── [현재] 나의 활동 (`/experiences`)
    │   ├── 진행 활동과 완료 경험 통합 목록
    │   ├── 과거 경험 작성 (`/experiences/new`)
    │   └── 완료 경험
    │       ├── 상세 (`/experiences/[id]`)
    │       ├── 수정 (`/experiences/[id]/edit`)
    │       └── AI 분석 (`/experiences/[id]/analysis`)
    └── [현재] CampusLog AI
        ├── AI 기반 활동 추천 (`/recommend`)
        └── 추천 기록 (`/recommend/history`)
```

`/login`과 `/signup`은 2차 MVP 목표 경로이며 현재 코드에 구현됐다는 뜻이 아닙니다. 비밀번호 재설정, 이메일 확인, 소셜 로그인과 계정 설정 경로는 세부 인증 정책이 확정될 때 별도로 정의합니다.

## 전역 내비게이션

### 로그인 사용자

- 데스크톱 순서: `오늘의 기록 → 나의 활동 → CampusLog AI`
- 모바일 축약: `오늘 → 활동 → AI`
- `/activities/**`는 `오늘의 기록`, `/experiences/**`는 `나의 활동`, `/recommend/**`는 `CampusLog AI`의 하위 위치로 표시합니다.
- 계정 메뉴와 로그아웃 위치는 공통 앱 셸 안에 추가하되 구체 배치는 디자인·UX 검토 후 확정합니다.

### 비로그인 사용자

- `/`의 3D 브랜드 표지는 v1.1 브랜드 진입 화면으로 유지합니다.
- 로그인과 회원가입으로 이동할 수 있는 명확한 진입점을 제공합니다.
- 비로그인 전용 내비게이션의 구성과 모바일 표현은 인증 화면 디자인과 함께 확정합니다.

## 화면 관계

### 오늘 기록과 활동 종료

```text
오늘의 기록 (`/dashboard`)
├── 활동 추가 → `/activities/new` → 저장 → 생성한 활동 상세
├── 진행 활동 선택 → `/activities/[id]`
└── 날짜 선택 → 한 일 작성 / 수정 / 삭제

진행 활동 상세 (`/activities/[id]`)
├── 오늘 한 일 기록하기 → `/dashboard`
├── 활동 종료 → AI 초안 검토 → 완료 경험 저장
└── 저장된 완료 경험 → `/experiences/[id]`
```

활동 추가 후에는 현재 v1.1 구현과 같이 생성한 활동 상세로 이동합니다. 2차 MVP에서 이 경로를 바꾸려면 USER_FLOW와 SCREEN_SPEC를 함께 갱신합니다.

### 나의 활동과 AI 분석

```text
나의 활동 (`/experiences`)
├── 진행 활동 → `/activities/[id]`
├── 완료 경험 → `/experiences/[id]`
└── 과거 경험 기록하기 → `/experiences/new`

완료 경험 상세 (`/experiences/[id]`)
├── 수정 → `/experiences/[id]/edit`
└── AI 분석 → `/experiences/[id]/analysis`
```

### CampusLog AI

```text
CampusLog AI (`/recommend`)
├── AI 기반 활동 추천
└── 추천 기록 (`/recommend/history`)
    └── 추천에 사용된 완료 경험 → `/experiences/[id]`
```

## 2차 MVP 인증 보호 정책

| 대상 | 정책 |
| --- | --- |
| `/` | 공개. 브랜드 진입 경험 유지 |
| `/login`, `/signup` | 공개. 로그인 상태에서 접근하면 기본적으로 `/dashboard`로 연결 |
| `/dashboard` | 로그인 필요 |
| `/activities/**` | 로그인 필요. 현재 사용자가 소유한 활동과 기록만 접근 |
| `/experiences/**` | 로그인 필요. 현재 사용자가 소유한 경험과 분석만 접근 |
| `/recommend/**` | 로그인 필요. 현재 사용자의 경험과 추천 기록만 접근 |
| AI·데이터 API | 유효한 서버 세션 필요. 요청 본문의 사용자 식별자를 신뢰하지 않고 서버에서 소유권 확인 |

- 보호 경로에 비로그인 상태로 접근하면 `/login`으로 보내고 원래 목적지를 안전한 `returnTo` 값으로 보존합니다.
- 인증 성공 후 `returnTo`가 유효하면 그 화면으로, 없으면 `/dashboard`로 이동합니다.
- 세션 확인 중에는 보호 데이터를 먼저 렌더링하지 않습니다.
- 다른 사용자의 ID나 존재하지 않는 ID로 접근했을 때 데이터 존재 여부가 과도하게 노출되지 않도록 동일한 접근 불가 처리를 사용합니다.
- 세션 만료 시 재로그인 이후 원래 화면으로 복귀할 수 있게 하며, 작성 중 데이터 보존 방식은 화면 상태·fallback 정책과 함께 확정합니다.

## 데이터 영역과 소유 관계

```text
User
├── TrackedActivity
│   ├── DailyLog
│   ├── ExperienceSynthesisDraft
│   └── generated Experience 관계
├── Experience
│   └── ExperienceAnalysis
└── RecommendationResult
```

- 활동 종료로 생성된 `Experience`는 원본 `TrackedActivity`와의 연결을 유지합니다.
- 미완료 합성 초안과 합성 상태는 원본 활동의 사용자에게 귀속하고 RLS를 적용합니다. 완료 경험 저장이 검증된 뒤 초안을 제거하며, 실패·재시도와 중복 생성 방지 상태는 원본 활동에 유지합니다.
- 모든 도메인 데이터는 로그인 사용자 소유로 저장하고 조회·수정·삭제 시 소유권을 검사합니다.
- AI 분석은 완료 경험, 추천 결과는 추천에 사용한 경험과 분석을 참조합니다.
- 실제 테이블, 키, cascade 삭제, 보존 기간과 감사 정보는 DB 설계에서 확정합니다.

## localStorage에서 DB로의 전환 위치 (Deferred / Optional)

현재 정식 범위에서는 신규 사용자가 계정별 DB부터 시작하므로 localStorage → 계정 DB 이전 화면을 필수 IA로 두지 않습니다. localStorage 원본은 자동 이전하거나 자동 삭제하지 않으며, 로그인 세션에서는 계정 DB 데이터를 기본으로 사용합니다.

향후 실제 기존 사용자 데이터 보존 요구가 생기면 아래 위치와 원칙으로 별도 설계합니다.

- 인증 직후 또는 첫 보호 화면 진입 시 현재 브라우저의 v1.1 데이터를 검사합니다.
- 이전 대상이 있으면 앱 셸 위의 전용 안내 화면, 모달 또는 단계형 패널로 이전을 안내합니다. 최종 표현은 디자인·UX 담당이 결정합니다.
- 이전 대상은 검증된 사용자 생성 데이터로 제한하고, 번들 샘플·개발 fixture·파싱 실패 항목은 대상 개수에서 제외합니다.
- 이전이 진행되는 동안 사용자가 중복 저장을 만들지 않도록 데이터별 상태를 명확히 표시합니다.
- 서버와 로컬 데이터가 함께 있을 때의 병합·중복·충돌 정책은 DB 스키마 확정 전 결정하지 않습니다.
- 마이그레이션 완료, 일부 실패, 전체 실패와 재시도 상태를 모두 지원합니다.
- 오프라인 임시 저장과 재동기화 여부는 기술 검토 후 확정하며, 결정 전까지 온라인 저장 성공을 가정하는 문구를 사용하지 않습니다.

## 개발 작업 경계

### 계정·데이터·AI 담당 영역

- 인증 모듈·server action·validation·오류 코드, 세션, 경로와 API 보호
- 사용자별 DB 모델, 저장·조회 API, localStorage 자동 이전·자동 삭제 금지 정책과 소유권 검증
- AI 분석·추천과 기존 활동 종료 합성의 입력·출력 계약, 모델 호출, 저장·보존과 오류 코드

### 디자인·UX 담당 영역

- 공개·인증·보호 영역을 포함한 공통 앱 셸과 내비게이션 경험
- `/login`, `/signup` route UI와 `components/auth/**` 표현 계층. Track A의 인증 contract를 사용하고 server action을 재정의하지 않음
- 기존 화면의 시각적 고도화, 반응형, 접근성, 빈 상태와 오류 피드백
- 로그인, 동기화와 AI 상태를 이해하기 쉽게 전달하는 화면 설계. 데이터 이전 UX는 Deferred / Optional

데이터 필드, API 응답, 경로 또는 인증 정책에 영향을 주는 변경은 두 작업 영역이 공동 검토합니다. 한 영역의 PR이 다른 영역의 구조를 임의로 확정하지 않습니다.

## 2차 MVP에서 아직 확정하지 않은 IA

- 이메일 확인과 비밀번호 재설정 경로
- OAuth / 소셜 로그인 provider와 경로
- 마이페이지 또는 계정 설정 화면
- 데이터 마이그레이션 UI 형식과 충돌 해결 화면
- 오프라인 모드 및 동기화 상태 화면
- AI 추천 후보 수, 비교 화면과 사용자 피드백 화면
- 파일 첨부, 관리자, 결제, 커뮤니티, 공개 프로필
