# CampusLog Design Guide

## 문서 상태

- 활성 계획 단계: 2차 MVP. 기능 구현은 v1.1 merge 후 시작합니다.
- 담당 Track: 디자인·사용자 경험 고도화
- 제품 기준선: v1.1
- 우선 문서: `docs/CURRENT_PHASE.md`, `PRD.md`, `docs/SCREEN_SPEC.md`
- 과거 1차 MVP 디자인 후보는 현재 기준이 아니며 `docs/archive/MVP_V1_1_BASELINE.md`에서 이력만 확인합니다.

## 1. 디자인 목표

CampusLog는 기록 도구의 신뢰감과 AI 기능의 명료함을 함께 갖춘 미니멀 SaaS를 목표로 합니다. 사용자는 화면을 해석하는 데 시간을 쓰지 않고 아래 행동을 자연스럽게 이어갈 수 있어야 합니다.

```text
오늘 한 일 기록
→ 활동의 누적 과정 확인
→ 완료 경험으로 정리
→ AI 분석
→ 지원 상황에 맞는 활동 추천
```

2차 MVP 디자인 고도화의 우선순위:

1. 로그인 전후 흐름이 끊기지 않는가
2. 저장 위치와 동기화 상태를 사용자가 이해할 수 있는가
3. 오늘의 기록, 나의 활동, CampusLog AI 사이 이동이 명확한가
4. AI 결과와 사용자 원본이 구분되는가
5. 모바일과 데스크톱에서 같은 작업을 완료할 수 있는가
6. 시각 효과보다 정렬, 간격, 글자 크기, 상태 표현이 먼저 다듬어졌는가

## 2. 브랜드와 시각 방향

### 브랜드

- 워드마크: `CampusLog`
- 워드마크 서체: Petrona 800, 촘촘한 자간
- 본문과 UI: Pretendard Variable
- 제품 성격: 차분함, 신뢰감, 기록의 지속성, 정돈된 AI

### 색상

현재 제품 코드의 검정·차콜·웜그레이 체계를 사용합니다.

- 브랜드 강한 색: `#1f1e1c`
- 브랜드 기본 색: `#343331`
- 주요 텍스트: `#241f1a`
- 보조 텍스트: `#5f5b54`
- 약한 텍스트: `#858077`
- 표면: 흰색과 near-white
- 경계: 낮은 채도의 얇은 neutral border
- focus: 브랜드 색을 투명하게 사용한 3px ring

캠퍼스 그린·민트는 현재 활성 브랜드 팔레트가 아닙니다. 상태 표현에 색상이 필요해도 다색 UI를 만들지 않고 중립색과 명도 차이를 먼저 사용합니다.

### 표면과 공간

- 넓은 여백과 명확한 정렬
- 작은 입력·버튼 8~12px radius
- 큰 작업 표면 18~28px radius
- 얇은 border와 낮은 강도의 shadow
- 카드 안에 카드가 반복되는 중첩 구조 최소화
- 장식용 그라디언트와 과도한 glass effect 금지

### 모션

- 상태 이해에 필요한 120~280ms 전환만 사용
- 목록 재배치, 상세 패널 등장, 로딩 피드백 중심
- 반복적인 부유, 반짝임과 목적 없는 장식 애니메이션 금지
- `prefers-reduced-motion`에서는 즉시 전환하거나 최소 duration 사용
- 결과가 발생하는 핵심 CTA는 Animate UI Ripple Button을 참고해 누른 위치에서 600ms 이내의 ripple과 짧은 hover·tap scale을 제공합니다. 키보드 활성화는 중앙에서 시작하고 reduced motion에서는 ripple과 scale을 생략합니다. 인증, 캘린더 날짜, 탭, 목록, 메뉴, 단순 링크, 닫기·삭제·숨기기에는 적용하지 않습니다.
- 비로그인 `/`는 기록 대상을 바꾸는 순환 문구와 스크롤 인증 영역을 사용하고, 3D 노트는 인증된 사용자의 제품 진입 화면으로만 유지
- 순환 문구는 `대학생활을`부터 시작해 `공모전을 / 해커톤을 / 프로젝트를 / 대회를` 순서로 전환하며 네모난 배경·테두리 없이 텍스트만 표시
- 순환 문구 아래에는 별도 `일시정지 / 재생` 컨트롤을 두지 않습니다. reduced motion에서는 첫 문구를 고정해 자동 전환을 시작하지 않습니다.
- 큰 주제인 `대학생활`은 브랜드 검정과 강한 굵기를 유지하고, 나머지 순환 명사는 웜그레이와 한 단계 가벼운 굵기로 위계를 구분. 조사 `을/를`은 모든 항목에서 브랜드 검정·강한 굵기를 고정
- 명사와 조사는 별도 motion 단위로 구성해 `을 → 을` 또는 `를 → 를`처럼 조사가 같을 때는 조사 DOM을 유지하고, `을 ↔ 를`로 실제 값이 바뀔 때만 짧고 약한 조사 전환을 적용
- 명사 전환은 기존 단어가 약 180ms 동안 폭을 유지한 채 사라지고, 새 단어는 100ms 뒤부터 약 280ms 동안 겹쳐 들어와 명사가 비는 프레임을 만들지 않습니다. 150ms 뒤부터 실제 렌더링 폭을 약 440ms 동안 연속 보간하며, 명사와 조사 사이에는 화면 크기에 따라 2~5px 여유를 둡니다. 새 단어는 조사 방향에 맞춰 정렬하고 글자 마스크에는 좌우 여유를 둬 마지막 획을 자르지 않으며 문장 중심은 전체 전환 동안 유지
- 첫 화면 제목은 스크롤 진행에 맞춰 위로 이동·축소·blur 되며 사라지고, reduced motion에서는 단어 순환과 스크롤 아웃을 모두 중지
- 첫 viewport에서 아래 방향의 작은 휠 입력이 누적되면 인증 viewport로 한 번에 부드럽게 이동합니다. 인증 화면의 위 방향 휠은 자동으로 첫 viewport까지 복귀시키지 않고 브라우저 기본 스크롤로 충분히 올려야 돌아가도록 하며, 인증 폼 아래의 일반 스크롤도 가로채지 않습니다. reduced motion에서는 아래 방향 이동만 즉시 처리
- 첫 viewport 하단 링크는 `스크롤하여 로그인 또는 회원가입` 한 줄과 화살표만 사용합니다. 별도 마우스 아이콘과 이중 문구 위계를 두지 않고, 화살표만 2.2초 주기로 최대 3px 이동하며 reduced motion에서는 완전히 정지
- 공개 진입 화면에서 장식용 반복 모션이나 강제 CSS scroll snap을 사용하지 않고, reduced motion과 200% 확대에서도 인증 폼 접근을 보장

### 프론트엔드 UI 기반

- Tailwind CSS v4는 `web/src/app/globals.css`와 `web/postcss.config.mjs`에서 구성합니다.
- shadcn/ui 설정은 `web/components.json`을 기준으로 하며 UI primitive는 `web/src/components/ui`, 공통 class 병합은 `web/src/lib/utils.ts`에 둡니다.
- 기존 v1.1 전역 스타일 회귀를 막기 위해 Tailwind theme·utilities만 불러오고 Preflight는 활성화하지 않습니다.
- 기존 전역 CSS 화면을 한 번에 교체하지 않고, 새 primitive와 화면부터 Tailwind utility를 적용해 회귀 범위를 제한합니다.
- Checkbox는 Animate UI Radix Checkbox 표현을 참고해 실제 선택 입력에만 사용합니다. default/accent와 sm/default/lg 변형, SVG check path, hover·press 피드백을 제공하며 키보드 Space, visible focus, native form 값과 reduced motion을 보존합니다. CopyButton은 아이콘 교체와 접근성 상태를 유지합니다.

## 3. 공통 앱 구조

### 데스크톱

- 좌측 앱 셸: CampusLog 워드마크와 핵심 메뉴
- 메뉴: 오늘의 기록 → 나의 활동 → CampusLog AI
- 좌측 하단: 얇은 구분선 아래 중립 표면 안에 원형 프로필 사진과 닉네임. 드롭다운은 프로필 식별과 로그아웃만 제공
- 주 콘텐츠: 흰색 작업 공간, 최대 읽기 폭 유지
- 상세가 열려도 원래 목록을 가능한 한 유지
- 활동·경험·AI의 작성·상세·수정·기록 같은 하위 경로에는 `상위 화면 > 현재 화면` 형태의 Basic Breadcrumb을 제목 앞에 표시합니다. 상위 항목만 링크로 제공하고 현재 항목은 `aria-current="page"`로 구분하며, 긴 이름 대신 안정적인 화면명을 사용해 모바일에서 줄바꿈 또는 말줄임으로 수용합니다.
- 인증된 제품 화면의 상위 경로도 `CampusLog > 현재 화면` Breadcrumb을 표시합니다. 모든 하위 화면은 `CampusLog`부터 기능 상위 경로와 현재 화면까지 전체 계층을 보여주며, AI는 `CampusLog > AI 기반 활동 추천 > 추천 기록`처럼 실제 이동 구조를 사용합니다. Breadcrumb의 `CampusLog`는 상단 워드마크와 같은 Petrona 800을 사용하고 나머지는 UI 서체를 유지합니다. 상위·하위 화면 모두 Breadcrumb을 H1 시작점 34px 위에 고정해 18px 높이 Breadcrumb과 H1 사이에 16px의 시각적 여백을 확보하고, 단계 수가 제목 위치를 바꾸지 않게 합니다. 랜딩·로그인·온보딩은 제외합니다.
- 활동 추가·상세, 경험 작성·상세·수정·분석, 추천 기록은 상위 화면과 동일한 최대 1120px 프레임을 사용합니다. 하위 화면 H1도 상위 화면 H1과 같은 좌표·크기·굵기·자간·행간에 고정하고 Breadcrumb은 H1을 밀어내지 않는 34px 위 보조 위치에 표시합니다. eyebrow나 상태 배지는 H1 위에 두지 않고 상태·보조 정보는 제목 아래에 배치하며, 우측 액션도 제목 위치에 영향을 주지 않습니다.

### 모바일

- 상단 앱 바: CampusLog와 `오늘 / 활동 / AI`
- 우측 계정 메뉴는 44px 원형 아바타로 축약하고 닉네임은 접근성 이름과 드롭다운 안에서 유지
- 터치 영역 최소 44px
- 목록 위, 상세 아래의 세로 흐름
- 하단 safe area와 고정 CTA 충돌 방지
- 긴 제목, URL, AI 문장이 가로 스크롤을 만들지 않음

### 상위 화면 제목 정렬

- `오늘의 기록`, `나의 활동`, `AI 기반 활동 추천`은 동일한 최대 1120px 페이지 프레임과 반응형 상단·좌우 여백을 사용합니다.
- 세 화면의 H1은 같은 크기·굵기·자간·행간을 사용하고 제목 아래 설명은 12px 간격과 동일한 본문 크기·행간으로 정렬합니다.
- `오늘의 기록` 날짜는 제목 위 별도 줄이 아니라 제목 옆 메타 정보로 표시해 화면 전환 시 H1 시작선이 달라지지 않게 합니다.
- 카드와 목록 너비는 각 화면의 정보 밀도에 맞게 유지하되 상위 제목 시작점은 공통 프레임의 왼쪽에 맞춥니다.

### 인증 전후

- 로그인 전: 좌측 상단에 Petrona 워드마크 `CampusLog`를 고정 표시하고, 첫 viewport의 `대학생활을 기록하다.` 순환 문구에서 다음 viewport의 중앙 로그인·회원가입 카드로 자연스럽게 연결
- 인증 영역: 좌우 소개 패널 없이 단일 카드만 중앙에 두고 같은 위치에서 로그인·회원가입 mode를 전환. 로그인 mode에서는 `Welcome back`과 제목 아래 소개 문구를 표시하지 않음
- 회원가입: 가입 방식 선택 뒤 이메일은 자격 증명, Google은 OAuth로 진행하고 두 방식 모두 이름 → 닉네임 Stepper를 완료
- 인증·프로필 성공 후: `/`의 3D 책 표지를 보여주며, 책을 선택하면 `/dashboard`로 진입
- 인증 후: 사용자가 원래 접근하려던 화면으로 복귀
- 세션 만료: 작성 내용을 잃지 않도록 안내 후 재로그인
- 로그아웃: 저장되지 않은 입력이 있으면 경고
- 인증 상태 확인 중: 전체 화면 깜빡임 대신 일관된 loading surface 제공

## 4. 핵심 화면 기준

### `/dashboard` 오늘의 기록

- 오늘 날짜와 `오늘의 기록` 제목을 가장 먼저 표시
- `현재 진행 중인 활동` 아래 활동 카드는 제목만 표시
- 시간 아이콘, 부가 설명, 체크 아이콘을 사용하지 않음
- 활동 추가 CTA는 `활동 추가`
- 활동 추가 CTA는 누른 위치에서 화면 가장자리 8px 안쪽의 near-white 대형 패널로 확장하고 28px 둥근 모서리를 유지합니다. 패널에는 `어떤 활동을 기록할까요?`와 필수 폼만 남깁니다. 닫힘에서는 밝은 표면색을 유지하면서 내용·표면을 첫 0.12초에 함께 페이드하고 배경까지 0.18초 안에 정리해 빈 사각형이나 잔여 dimming 없이 원래 CTA로 돌아옵니다. 스크롤 잠금 전후에도 배경 카드의 콘텐츠 폭을 바꾸는 별도 body padding을 적용하지 않습니다. 콘텐츠 왜곡 없는 열림 전환, 초점 가두기·복귀, 스크롤 잠금과 reduced motion을 제공합니다.
- `/dashboard`의 캘린더 옆에는 별도 `오늘 한 일 기록하기` 카드 대신 선택 날짜의 `날짜별 기록` 이벤트 목록을 표시합니다. 각 항목은 동일한 검정 세로 accent, 활동 제목과 기록 내용 한 줄을 기본 표시하며 hover와 `focus-within`에서 내용을 최대 3줄로 펼칩니다. 이 목록에서는 진행 중·종료 상태를 별도로 표시하지 않습니다. 항목 사이를 이동할 때 line-clamp를 즉시 교체하지 않고 동일한 3줄 제한 안에서 높이와 투명도만 ease-out으로 전환해 목록 재배치가 튀지 않게 합니다. 긴 내용은 말줄임 처리하고 전체 내용은 수정 패널에서 확인합니다.
- 날짜별 기록 헤더의 44px `+` 버튼으로 기존 중앙 작성 패널을 열며, 진행 중인 활동이 없을 때는 작성 화면을 바로 열지 않고 활동이 필요한 이유와 선택 가능한 `활동 추가` 액션을 담은 안내 dialog를 먼저 제공합니다.
- 플로팅 패널은 `한 일 남기기` 제목과 `어떤 일을 하셨나요?` 입력 라벨을 사용하고 활동 선택, 한 일 입력과 글자 수, 취소·저장을 한 흐름으로 제공합니다. 기록 패널은 데스크톱에서 최대 520px 너비를 사용하고 모바일에서는 좌우 14px 안전 여백 안에서 유동적으로 줄어들어 입력 문장의 가독성을 확보합니다. 활동 필드는 처음 열 때 특정 활동을 자동 선택하지 않고 `활동을 선택하세요` 안내를 표시하며, 검색 입력 없이 클릭 또는 방향키·Enter로 목록에서 하나를 선택합니다. 선택한 활동은 행 오른쪽의 체크 표시로 구분합니다. 필드에는 화살표만 유지하고 목록은 38px 높이의 조밀한 행과 작은 본문 글꼴을 사용합니다. 긴 활동명은 목록에서 최대 두 줄로 확인할 수 있고 한 일 입력은 최소 148px·5줄 높이로 시작해 가로·세로 읽기 비율을 확보합니다.
- 한 일 입력 도움말은 자세하게 적을수록 AI가 더 정확하게 분석할 수 있음을 안내
- 패널 닫기와 `Escape`는 작성 중 입력을 보존하고, 명시적인 `작성 취소`에서만 초기화. 저장 실패 시 패널과 입력을 유지하고 저장 성공 후에만 닫음
- 데스크톱과 모바일 모두 visual viewport·safe area 안의 화면 정중앙에 표시하고, 내용이 높이를 넘으면 내부 스크롤과 고정 footer로 저장 액션 접근을 보장
- 패널이 열리면 textarea로 초점을 이동하고 내부에 초점을 가두며, 닫힌 뒤 원래 CTA 또는 수정 버튼으로 복귀. reduced motion에서는 확장·이동 효과를 제거
- 캘린더는 기록 개수와 선택 날짜를 명확히 구분
- 캘린더 헤더는 연도와 월을 각각 선택하는 조밀한 Select를 먼저 배치하고 기존 이전·오늘·다음 이동을 함께 유지합니다. 연도 목록은 1980년부터 현재까지 역순으로 제공하고 현재 연도에서는 미래 월을 비활성화하며, 선택 체크·방향키·Enter·Escape와 reduced motion을 지원합니다.
- 캘린더 날짜 hover는 셀 위치나 크기를 움직이지 않고 배경·테두리·글자색만 자연스럽게 전환합니다.
- 캘린더 헤더는 중복되는 `월간 기록` 보조 문구 없이 현재 연·월과 탐색 버튼만 표시합니다.
- 미래 날짜는 비활성화
- 기록 작성·수정·삭제 결과를 인라인 상태로 안내

### `나의 활동` 검색

- 검색을 열어도 `나의 활동` 제목은 유지하고 공간 확보를 위해 활동 수와 `진행 중` 배지만 잠시 숨깁니다. 검색창은 같은 헤더 높이에서 제목 오른쪽의 남은 폭 안에서 최대 250px로 펼치며, 닫으면 같은 위치에서 축소되어 헤더가 위아래로 재배치되지 않게 합니다.

### 로그인 / 회원가입

- 로그인과 회원가입을 별도 라우트 또는 명확한 mode로 제공
- Aceternity signup form의 중립 표면·입력 위계·full-width CTA를 참고하되 실제 Supabase action과 상태 contract를 유지
- 회원가입 첫 화면은 이메일과 Google 두 선택만 보여주고, 이메일 선택 시에만 이메일·비밀번호를 표시
- 이름·닉네임은 React Bits Stepper의 진행 표시와 방향 전환을 절제해 적용하되 별도 중첩 카드, 보라색 강조, 큰 이동 모션은 사용하지 않음
- Stepper indicator는 상태 표시용 `ol`로 제공하고 단계 이동은 이전·다음 버튼으로만 제어. 단계 전환은 22px 이하·240ms 이하이며 reduced motion에서는 즉시 전환
- 필수 입력만 요청하고 이름·닉네임의 비공개 사용 목적을 설명하며 학교 인증 등 미승인 입력은 추가하지 않음
- 비밀번호 요구사항과 오류를 제출 전에 이해할 수 있게 표시
- 필드 형식 오류, 인증 실패, 요청 제한, 네트워크 오류를 구분하되 이메일 등록 여부를 추측할 수 있는 문구는 사용하지 않음
- 비밀번호 요구사항은 Supabase 설정과 같은 공유 validation contract에서 가져와 폼과 서버가 다르게 안내하지 않음
- 이전·다음 이동에서 입력을 보존하고 저장 실패 시 Stepper를 닫지 않으며, 첫 invalid 입력으로 초점을 이동
- 성공 후 사용자가 작업을 시작할 다음 화면을 명확히 제공
- 소셜 로그인은 실제 도입이 승인된 provider만 표시

### 데이터 이전 안내

- 브라우저에 이전 가능한 v1.1 localStorage 데이터가 있으면 서버 계정 데이터 유무와 관계없이 이전 안내 제공
- 서버 데이터가 있으면 자동 병합하지 않고 중복·충돌 확인 단계로 연결
- 번들 샘플·개발 fixture·파싱 실패 항목은 이전 가능 개수에서 제외
- `이 계정으로 가져오기`, `브라우저 기록 유지`, `나중에` 같은 선택지는 실제 정책 확정 후 표시
- 이전 대상, 중복 처리, 삭제 여부를 행동 전에 설명
- 진행률, 완료, 부분 실패, 재시도 상태 제공
- 자동 삭제나 조용한 덮어쓰기 금지

## 5. AI UX 원칙

- AI가 생성한 문장과 사용자가 직접 작성한 원본을 구분합니다.
- 생성 근거와 사용된 활동을 확인할 수 있게 합니다.
- 분석·추천이 실패해도 원본 경험과 입력 문항을 유지합니다.
- 같은 요청의 중복 실행을 방지하고 loading 중 버튼을 disabled 처리합니다.
- 저장 성공 여부를 확인한 뒤 성공 화면으로 전환합니다.
- 사용자가 AI 결과를 수정, 복사, 다시 생성할 수 있는 범위를 명확히 합니다.
- 추천 정확도 고도화가 여러 후보를 포함한다면 후보 수와 비교 방식은 PRD에서 승인한 뒤 UI를 설계합니다.

## 6. 상태 설계

모든 핵심 화면은 다음 상태를 가집니다.

- initial
- loading
- empty
- success
- validation error
- network / server error
- permission error
- partial success
- retrying

인증·DB 화면은 추가로 다음 상태를 고려합니다.

- session checking
- unauthenticated
- authenticated
- session expired
- offline / reconnecting
- local data migration available
- migration in progress / failed / completed

색상만으로 상태를 전달하지 않고 텍스트, 아이콘 또는 구조를 함께 사용합니다. 단, 단순 선택 태그처럼 라디오 semantics가 이미 제공되는 경우 체크 아이콘 없이 색상 변화로 시각 상태를 표현할 수 있습니다.

## 7. 입력과 버튼

- label과 input의 연결을 유지
- 필수 / 선택 여부를 일관되게 표시
- 입력 중 값을 임의로 삭제하지 않음
- 오류 발생 시 첫 오류로 초점을 이동하되 사용자가 입력하던 내용 보존
- Primary CTA는 화면당 하나를 원칙으로 함
- destructive action은 기본 CTA와 분리하고 확인 제공
- disabled 상태만으로 이유를 숨기지 않고 필요하면 설명 제공
- 버튼 문구는 결과 중심으로 작성: `저장`, `로그인`, `회원가입`, `다시 시도`

## 8. 반응형과 접근성

- 모바일 기준 390×844에서 핵심 흐름 검증
- 데스크톱 기준 1280×720 이상에서 정보 밀도 검증
- 200% 확대에서도 핵심 조작 가능
- 키보드만으로 메뉴, 폼, 목록, 상세, Dialog 조작 가능
- focus-visible을 제거하지 않음
- Dialog의 focus trap과 닫기 후 초점 복귀
- 텍스트 대비 WCAG AA 수준 목표
- 상태 배지와 작은 텍스트는 색상 대비 별도 확인
- aria-live는 저장·AI 결과처럼 중요한 비동기 상태에 제한적으로 사용
- skeleton은 실제 레이아웃과 유사하게 구성하고 무한 로딩에 fallback 제공

## 9. Track 협업 규칙

### 디자인·UX Track

- API response shape와 DB schema를 임의로 바꾸지 않음
- 임시 mock 상태는 공개 가능한 가짜 데이터만 사용
- 디자인 변경이 사용자 흐름을 바꾸면 USER_FLOW / IA / SCREEN_SPEC를 함께 수정

### 인증·데이터·AI Track

- UI에 필요한 loading / empty / error / permission 상태 계약을 제공
- 사용자 흐름을 바꾸는 redirect, 권한, migration 정책은 문서와 함께 변경
- backend error를 그대로 노출하지 않고 사용자용 오류 code / message 계약 제공

### 공통

- 같은 컴포넌트를 동시에 수정할 때 담당 PR과 merge 순서 합의
- 디자인 token과 API contract 변경은 별도 설명 없이 섞지 않음
- 현재 문서 본문을 갱신하고 하단에 override 섹션을 계속 추가하지 않음

## 10. 디자인 완료 조건

- 최신 검정·차콜 제품 양식이 모든 제품 화면에서 일관됨
- 로그인 전후와 localStorage → DB 전환 흐름을 사용자가 이해할 수 있음
- 오늘의 기록 → 나의 활동 → CampusLog AI 흐름이 끊기지 않음
- 핵심 화면의 loading / empty / error / success 상태가 구현됨
- 모바일·데스크톱 가로 overflow 없음
- 키보드, focus, 대비, reduced motion 검증 완료
- 과한 시각 효과 없이 정렬, 간격, 타이포그래피 품질이 확보됨
- 실제 API / schema 상태와 화면 문구가 일치함
